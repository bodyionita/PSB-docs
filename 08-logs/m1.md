## Task 3 ‚Äî durability (three slices)

The durability task was built in three reviewable slices (all committed locally, **not pushed**).

### Slice A ‚Äî git-backed `VaultBackupService` (`884855f`)

Replaces the `LoggingVaultBackup` no-op stub the pipeline depended on with the real service ‚Äî the
**one owner of all vault git ops behind a single `asyncio` lock** (ADR-014 ¬ß2‚Äì3):

- **`GitRepo`** (`services/git_repo.py`) ‚Äî thin subprocess wrapper (async via `to_thread`) behind a
  **`GitClient` protocol** (fakeable, mirrors the `CaptureStore` seam). Only ever **fast-forward**
  pushes (`git push HEAD:branch`, no `--force`); the sole reconciliation is a **merge** (`pull
  --no-rebase --no-edit`) ‚Äî never rebase/force/reset. `is_merging`/`abort_merge` guards; push/pull
  **network timeouts are soft failures** (commits kept local, retried).
- **`VaultBackupService`** ‚Äî **debounced ~60s batch commits** (coalesced, deduped message); every
  stage/commit/push serialises under one lock; **gc/reflog pins**
  (`gc.pruneExpire`/`reflogExpire`/`reflogExpireUnreachable=never`, `gc.auto=0`) + commit identity
  applied **idempotently at startup**; **empty-repo bootstrap** (`Inbox/`, `Summaries/Daily|Weekly/`,
  plane folders + `.gitkeep`, the ADR-014 ¬ß3 `.gitignore` ‚Äî excludes only Obsidian UI cache + OS
  cruft, never notes/`.trash` ‚Äî initial commit + `push -u`; subsumes the `PSB-vault` initial-commit
  follow-up); **heal-on-reject** (non-ff push ‚Üí pull-merge ‚Üí re-push, healing GitHub if a client
  rewound it); best-effort push (unreachable remote ‚áí commits stay local, next backup reconciles ‚Äî ¬ß5).
- **`POST /admin/backup`** ‚Üí `{committed, pushed}` (03-api), session-gated; forces an immediate commit
  + push, folding in any pending debounced reasons.
- **Lifespan** ‚Äî builds the service, `ensure_ready()` at boot (init if needed ‚Üí pins ‚Üí bootstrap),
  wires it into `CapturePipeline` (replacing the stub), and on shutdown **drains the pipeline ‚Üí
  flushes the backup ‚Üí disconnects the DB** (order matters: a draining capture may enqueue a commit).
  New non-secret settings (`vault_git_remote/branch`, `vault_backup_debounce_seconds`,
  `git_user_name/email`) in `.env.example`.

- **Independent review** (fresh agent, diff vs ADR-014 + CLAUDE.md rules): **1 must-fix resolved** ‚Äî
  cancelling the debounce timer *mid-commit* (via `flush`/`backup_now`) raised `CancelledError` into
  the in-flight batch, releasing the lock while the uncancellable `to_thread` git subprocess kept
  running, so the canceller launched a **second concurrent git command** (broke the single-lock
  invariant). Fixed by **`asyncio.shield`ing the commit phase** so the canceller blocks on the lock
  until the in-flight batch finishes; **regression test verified it fails without the shield**. 3
  minors also applied: an in-progress-merge commit guard (never capture conflict markers),
  reschedule-on-failed-batch (don't wait for the nightly sweep), and soft push/pull timeouts.
- **Verification:** 89 server unit tests + `ruff` green. Orchestration (debounce/coalesce,
  heal-on-reject success+abort, no-remote, no-op flush, backup_now fold-in, the shield regression)
  tested with a `FakeGitRepo`; **real-git integration** (bootstrap‚Üípush to a bare remote, a genuine
  non-ff heal-merge preserving both sides) tested against actual git, **skipped when git is absent** so
  CI stays green; `create_app()` imports clean and `/admin/backup` registers under `/api/v1`.

### Slice B1 ‚Äî R2 job layer (`7f3c4a7`)

The four R2/WORM jobs + `agent_runs` writer + CLI. Driven for now by the CLI; the in-process
scheduler + `/health` 4th leg are Slice B2.

- **`ObjectStore` seam + `R2ObjectStore`** (`services/object_store.py`) ‚Äî Cloudflare R2 via the S3
  API; **boto3 imported lazily** inside the impl so the module graph / tests / CI never pull it.
  `build_object_store(settings)` ‚Üí `None` when creds are absent (dev) ‚áí the R2 jobs record a
  **skipped** run. New R2 settings
  (`r2_account_id/access_key_id/secret_access_key/bucket/endpoint_url`; endpoint derives
  `https://<account>.r2.cloudflarestorage.com`) in `.env.example`.
- **`AgentRunStore`** (`services/agent_runs.py`, plain SQL, no ORM) ‚Äî every job **opens/closes an
  `agent_runs` row** (`running` ‚Üí `succeeded`/`failed`/`skipped`) with a summary + `details` jsonb
  (vision P8); `latest(agent, status=)` feeds the drill and (Slice B2) `/health`.
- **Git bundle support** ‚Äî `GitRepo` gains `bundle_all` / `commit_count` / `tracked_file_count` /
  `verify_bundle` / `clone_from`; `VaultBackupService` gains a **`Fingerprint`** (HEAD sha + monotonic
  commit count + file count) with `write_bundle` / `snapshot_fingerprint` **both under the one git
  lock** (live-repo git never runs outside the lock, ADR-014 ¬ß2).
- **`BackupJobs`** (`services/backup_jobs.py`) ‚Äî the four jobs + the 04:55 sweep: **`vault-backup`**
  (`git bundle --all` ‚Üí R2 WORM + `.manifest.json`, records the fingerprint), **`integrity-drill`**
  (download the last *good* bundle ‚Üí `git bundle verify` + clone ‚Üí assert fingerprint vs manifest +
  **monotonic** live count), **`db-backup`** (`pg_dump` ‚Üí R2), **`data-sync`** (`DATA_PATH` raw inputs
  ‚Üí R2). Never-crash (rule 7): each ends as a `failed` run with context; all blocking IO
  (boto3/pg_dump/git/file/tempdir) via `asyncio.to_thread`. The sweep is a plain `backup_now` (the git
  commit), not one of the four named runs.
- **CLI** (`app/cli.py`) ‚Äî `python -m app.cli {vault-backup|integrity-drill|db-backup|data-sync|
  vault-sweep}` for a future external scheduler (no rework); `ensure_ready()` first. `boto3` added to
  deps (prod only).

- **Independent review** (fresh agent, diff vs ADR-014 ¬ß1/¬ß6/¬ß7 + CLAUDE.md rules): **1 must-fix
  resolved** ‚Äî the **monotonic truncation alarm wasn't actually enforced**: the drill only compared
  live vs the *latest* bundle, so a persistent commit-count DROP got silently re-recorded as the new
  baseline and never alarmed. Fixed by failing the **`vault-backup`** run when its count drops below
  the last **succeeded** bundle (so the good baseline is preserved) **and** drilling the last
  *succeeded* bundle ‚Äî a truncation now always alarms. 3 minors applied: wrap the `agent_runs` row-open
  so a DB-down never crashes a job (rule 7); CLI `ensure_ready()` + a use-CLI-**or**-scheduler note; a
  note that the drill's live-repo check stands in for the "and GitHub" side. **Logged follow-ups:** an
  independent GitHub-side fingerprint fetch; data-sync re-uploads the whole corpus nightly with
  flattened keys (fine for the flat uuid-named inputs today).
- **Verification:** 105 server unit tests + `ruff` green. Jobs tested with fake R2/agent-runs/git
  (fingerprint upload, monotonic-regression ‚Üí failed run, skip-when-disabled, drill pass + regression +
  mismatch + no-bundle, db/data uploads, row-open-failure resilience); **real-git** bundle‚ÜíR2‚Üíverify‚Üí
  clone‚Üídrill **round-trip** against actual git (skipped when git absent); CLI arg-handling + job
  wiring smoke; `boto3` confirmed **not** imported without creds.

### Slice B2 ‚Äî scheduler + `/health` backups leg (`1e3c420`)

The final durability piece ‚Äî the in-process APScheduler and the `/health` 4th leg. Closes the
durability task (Slices A + B1 + B2). Delivered:

- **`BackupScheduler`** (`services/scheduler.py`) ‚Äî wraps APScheduler's `AsyncIOScheduler`; the
  job‚Üíschedule map is **pure data** (`job_specs()`, so it unit-tests without a loop),
  `start()`/`shutdown()` are the only parts touching APScheduler. Registers the **five** durability
  jobs on crontab windows in `scheduler_tz`, staggered inside the ADR-010 03:00‚Äì05:00 band and clear
  of the reserved M4 slots (Slack 03:00 / rescan 03:40 / summary 04:10 / review 04:40): `data-sync`
  03:10, `db-backup` 03:25, `integrity-drill` Sun 04:30, **`vault-sweep` 04:55** (ADR-010),
  **`vault-backup` bundle 04:57** (WORM right after the sweep, so the bundle includes it ‚Äî ADR-014 ¬ß3
  "checkpoint = commit+push+bundle"). `misfire_grace_time` (default 1h) + `coalesce` +
  `max_instances=1` realise ADR-010's "missed window ‚áí next night covers it" (a long outage skips,
  restart jitter still fires, a job never overlaps itself on the 4GB VPS).
- **Lifespan wiring** (`main.py`) ‚Äî when `enable_scheduler` (off by default; exactly one prod instance
  sets it), builds `BackupJobs` via the shared **`build_backup_jobs`** factory (moved from `cli.py`
  into `backup_jobs.py` so CLI and scheduler share one constructor) and `start()`s it inside the
  lifespan event loop; on shutdown **stops the scheduler first** (a job may enqueue a vault commit),
  then drains the pipeline ‚Üí flushes the backup ‚Üí disconnects the DB.
- **`/health` 4th leg** (`system_health.py`, `models.py`, `routers/health.py`) ‚Äî `backups` degrades

  `/health` (rule 7). **Prod-gated exactly like `git_remote`** (reported always, gates `ok` only in
  prod). Contract already recorded in [03-api](../03-api.md).
- **Settings** (rule 9 ‚Äî no hardcoded schedules): five crontab strings +
  `scheduler_misfire_grace_seconds` + `integrity_drill_max_age_days`, all in `.env.example`.

- **Independent review** (fresh agent, diff vs Slice B2 spec + ADR-010/014 ¬ß6 + 03-api + CLAUDE.md
  rules): **no must-fix.** Verified: all five jobs scheduled to the right coroutines; times in-window,
  staggered, sweep@04:55, bundle-after-sweep; misfire/coalesce/max_instances consistent with ADR-010;
  lifespan start/shutdown ordering correct + `enable_scheduler`-gated; backups-leg prod gating mirrors
  `git_remote`; **no TZ bug** (`agent_runs.started_at` is `timestamptz` ‚áí asyncpg returns aware UTC);
  DB-read error degrades, never crashes `/health`; all schedules/thresholds are settings. One minor

  `shutdown(wait=False)` doesn't await an in-flight backup job, so a job finishing after
  `db.disconnect()` could log an error on the closed pool ‚Äî background task, idempotent, service never

- **Verification:** **116 server unit tests + `ruff` green** (was 105; +11: scheduler job-spec mapping

## Live-Accept bugs (2026-07-13)

**Audio not durable across deploys (rule-2 violation; fixed).** A voice capture from before a redeploy
could not be retried afterwards. Root cause: `data_path` defaults to the dev-relative `../data` and
**prod never set `DATA_PATH`**, so the container wrote audio to its *ephemeral* `/data` (WORKDIR `/app`
‚Üí `/app/../data`), **not** the persistent `/srv/data` bind mount ‚Äî so a redeploy wiped un-processed
audio and retry found no file. The R2 audio-sync job reads the same `data_path`, so it was backing up
the wrong (ephemeral) directory too. Fix: pin **`DATA_PATH=/srv/data`** in `deploy/defaults.env` (+
`.env.example`). **Requires a redeploy to take effect**; audio captured before the fix on the ephemeral
path is unrecoverable if already wiped. Once deployed, audio persists on the R2-synced `/srv/data`
volume and survives redeploys, so retry works across deploys (matches the recovery matrix in
02-data-model).

**`GROQ_API_KEY` never reached the container (fixed).** Voice transcription failed with `groq: no API
key configured; openai ‚Ä¶ 429`: the STT chain worked (Groq primary ‚Üí no key ‚Üí fell back to a
rate-limited OpenAI), but the container's `GROQ_API_KEY` was empty despite the GitHub Actions secret
existing. Root cause: the CI deploy workflow's `.env` render step listed
`OPENAI_API_KEY`/`NEBIUS_API_KEY`/‚Ä¶ but **not `GROQ_API_KEY`**, so it was never written to `deploy/.env`
(07-infrastructure already listed it as a rendered secret ‚Äî the workflow had drifted). Fix: add
`GROQ_API_KEY` to the render `env:` block + the `printf` in `.github/workflows/ci.yml`. **Requires a
redeploy.** Once rendered, Groq (free tier) is the STT primary and voice no longer depends on the 429'd
OpenAI.
