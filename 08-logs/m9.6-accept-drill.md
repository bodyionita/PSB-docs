## A. Composite compose (phone) — the core drill

1. [ ] Capture tab → compose one capture with **text + ≥2 photos + a voice note** (record orb).
2. [ ] **Remove** one part with its 'x', confirm the tile disappears, then **re-add** it.
3. [ ] Empty the draft to **0 parts** → confirm **Send is disabled**.
4. [ ] Try recording a **2nd voice** → confirm it's **refused** (client blocks; server also enforces ≤1).
5. [ ] Re-add parts, **Send** → exactly **one** capture appears in Recents (not one per part).
6. [ ] Open the capture / wait for processing → confirm **node(s) cross-reference parts**: a node
       reflects **both** its photo and the voice narration (blended organize, not per-part silos).
7. [ ] Confirm **per-node attribution**: each node shows **only its attributed** media; an
       **unattributed** part stays on the **capture only** (not forced onto every node).

**Post-drill DB check** (this is the first real multi-part composite — nothing multi-media existed
in prod before, so this is what proves part-ordinals + `parts:[…]` attribution end-to-end). In the
Supabase SQL editor, using the capture id from the drill:
```sql
-- parts get sequential ordinals (1..N), voice + photos both present
select part_ordinal, kind, status from media where capture_id = '<CAPTURE_ID>'
 order by part_ordinal nulls last, created_at;
-- per-node attribution: each node links only the parts it named
select n.title, array_agg(m.part_ordinal order by m.part_ordinal) as attributed_ordinals
  from node_media nm join media m on m.id = nm.media_id join nodes n on n.id = nm.node_id
 where m.capture_id = '<CAPTURE_ID>' group by n.id, n.title;
```
PASS: ordinals are 1..N (no NULLs on the new parts); at least one node links a strict subset (not
all-to-all); an unattributed part appears in `media` but in **no** `node_media` row.

## B. Draft resume / discard (phone)

1. [ ] Start a draft, attach text + ≥1 part, **close the app** (kill it).
2. [ ] Reopen → the draft **resumes** (text + parts intact).
3. [ ] Hit **Discard** → the draft is dropped; a fresh compose starts empty.

DB sanity: `select count(*) from captures where status='draft';` should return to the count of any
*other* intentionally-open drafts (a stale empty draft, id `6434…`, existed at drill prep — GC is
7 days, so it may still be there; discarding yours shouldn't leave an extra).

## D. reprocess-all byte-parity (VPS CLI)

The durability invariant: everything derived rebuilds from the store byte-identically (ADR-042).
```sh
# On the VPS, from the compose dir:
docker compose exec api python -m app.cli reprocess-all
```
1. [ ] Before: snapshot the composite's node set + media links (queries below).
2. [ ] Run `reprocess-all`; wait for completion.
3. [ ] After: re-run the snapshot → **node set + `node_media` are stable** (same nodes, same
       part→node links). The cached `raw_text` (text_body + ordinal-ordered `[[part N · kind]]`
       markers) is what makes the replay deterministic.
```sql
-- snapshot (run before + after; diff the outputs)
select id, title from nodes where merged_into is null
  and id in (select node_id from node_media nm join media m on m.id=nm.media_id
             where m.capture_id='<CAPTURE_ID>') order by title;
select m.part_ordinal, nm.node_id from node_media nm join media m on m.id=nm.media_id
 where m.capture_id='<CAPTURE_ID>' order by m.part_ordinal, nm.node_id;
```
