## Task 1

**Model routing engine** ([ADR-025](../adr/025-ui-editable-model-routing-and-per-task-effort.md) +
[ADR-043](../adr/043-quick-routing-tier-m4.md)) — server, local, **not pushed**.

**New `ModelRoutingService`** (`server/app/services/model_routing.py`) — the routing brain over
**three groups** (`chat`/`conspect`/`quick`), modeled identically per ADR-043 §1. Each group seeds
from config (`chat_chain`/`distill_chain`/`quick_chain` + `claude_max_effort`/`quick_effort`) and is
overlaid by user-saved routing in `app_settings` under one `model_routing` jsonb key
(`PgModelRoutingStore`, plain SQL over asyncpg — rule 5/ADR-011). In-memory cache **busted on save**
(`invalidate()`), so edits apply next request with no restart (ADR-025 §3). `resolve(group)` returns
a `RoutingDecision{chain, effort_by_provider}`: chain = `[active, fallback]` filtered to registered
chat providers, **degrading to the config seed chain (+ its effort) when a saved id is unknown/stale**
(rule 7); effort filtered to effort-capable providers only. `complete(group, messages,
requested_model=?)` delegates to the registry. `build_model_routing(settings, db, registry)` factory
shared by `main.py` + the CLI factories.

**Per-call effort through the provider boundary** (ADR-025 §4): `ChatProvider.complete` gains an
`effort` arg + a `supports_effort` class flag (`ClaudeMaxProvider` True → `effort or self._effort`;
`OpenAICompatibleProvider`/Nebius accepts-and-ignores, flag stays False). The registry threads
per-provider effort in `_chat_over_chain` and exposes a new `run_chain(messages, *, chain,
effort_by_provider, requested_model)` (the mechanics the service drives) plus `supports_chat` /
`supports_effort` accessors (so the service + a future GET /settings never hardcode which models
take effort). `registry.chat`/`distill` now delegate to `run_chain` — **behavior-preserved, the 21
registry-fallback tests stay green** (+1 new: `build_registry` registers `claude-max-sonnet`).

**`claude-max-sonnet` provider instance** (ADR-043 §3): a second `ClaudeMaxProvider` over the SAME
`claude` CLI with a config-driven `claude_max_sonnet_model` (default `claude-sonnet-4-6`; the exact
CLI alias is a one-line config swap confirmed at deploy), effort-capable, constructor effort seeded
to the cheap-tier `quick_effort`. `quick` seed = `[claude-max-sonnet, nebius]` @ low.

**6 conspect call sites rewired** through `routing.complete("conspect", …)`: capture organize +
nudge (`services/capture_pipeline.py`), entity resolution (`entities/resolver.py`), profile
generation (`entities/profile_refresh.py`), tag consolidation (`tags/service.py`), edge
consolidation (`vocab/edge_consolidation.py`). Services that still need the registry for a
non-routed leg keep it (profile-refresh embed, capture-pipeline transcribe); the three distill-only
services dropped it. Wired in `main.py` (`app.state.model_routing` + `get_model_routing` dependency)
and the standalone CLI factories (`services/reprocess.py`, `entities/profile_refresh.py`).

**No migration** — routing lives in the existing `app_settings` jsonb (ADR-043 §Consequences).

**Verification:** `ruff check` clean; **396 tests green** (+25: 13 new `test_model_routing.py` seed/
override/effort/degradation/cache-bust cases against fakes, +1 registry wiring test, +11 service
tests migrated to the routing seam). New unit tests use a `FakeModelRoutingStore` + `fake_routing`
helper (no live DB/LLM — 08 policy).

**Independent review — APPROVE, no must-fix** (fresh agent vs ADR-025/043 + CLAUDE.md rules + diff):
verified 3-group symmetry, effort routed to Claude tiers only, all 6 sites rewired with no residual
`registry.distill()` in app code, rule-7 degrade, no circular imports, CLI factories updated. Three
minors, **all fixed**: (1) sonnet constructor effort → `quick_effort` (was `claude_max_effort`);
(2) rule-7 degrade now also falls back to the seed's effort (effort read from whichever routing
supplied the chain); (3) added the `build_registry` wiring test.

## Task 2

**Retrieval — hybrid vector+FTS RRF + recency prior + temporal filters** ([ADR-032](../adr/032-prior-art-adoptions.md)
§5/§7 + [ADR-037](../adr/037-profile-embedding-in-search-m3.md); 03-api §Search) — server, local, **not pushed**.

**Migration 008** (`008_fts_tsvector.py`): a generated **`tsvector`** column on **both**
`chunks` (`to_tsvector('english', content)`) and `node_profiles` (`… profile`), each `GENERATED
ALWAYS AS … STORED` + a **GIN** index (`chunks_tsv_gin` / `node_profiles_tsv_gin`). The FTS leg
mirrors the ADR-037 vector union so RRF fuses the **same** `chunks ⊍ node_profiles` node universe.
Generated + store-derived ⇒ no writer touches `tsv`; `POST /admin/reindex` and reprocess rebuild it
for free (Rule 1). `'english'` matches the asserted English corpus, so a non-English `tsquery`
self-suppresses (no matching lexemes) with **no language-detect dependency**. `down_revision=007`,
idempotent (`IF NOT EXISTS`), reversible downgrade; runtime head-check test moved 007→**008**.

**Hybrid `PgSearchStore.search_chunks`** (rewritten): two legs, each a best-per-node union over
`chunks ⊍ node_profiles`, bounded to a **candidate pool** (`LIMIT candidates`) before fusion —
- **VEC**: cosine `<=>` over `chunks.embedding ⊍ node_profiles.embedding`, `DISTINCT ON (node_id)`
  nearest row per node;
- **FTS**: `tsv @@ websearch_to_tsquery('english', query)` over `chunks.tsv ⊍ node_profiles.tsv`,
  best `ts_rank` per node.
Each leg is `row_number()`-ranked (deterministic `, node_id` tiebreak) and **`FULL OUTER JOIN`**ed
on `node_id`; **RRF** fuses by **rank, never raw score** (`1/(k+vec_rank) + 1/(k+fts_rank)`, `k=60`
— cosine and `ts_rank` are incommensurate, ADR-032 §5). A **mild recency prior** then multiplies the
fused score (bounded `[floor,1]`, `floor + (1-floor)·0.5^(age/half_life)` on `occurred_start ??
node_created_at`, future dates clamped to age 0, UTC-pinned `today`) **pre-cut** (ADR-032 §7). The FTS
leg self-suppresses on non-English / zero-lexeme queries (0 matching rows → vector-only ranking).
`websearch_to_tsquery` (not `to_tsquery`) never raises on arbitrary user text; params bound (no
injection). `min_score` now floors the **fused RRF×recency** score (default `0.0` ⇒ backward-safe;
scale note logged for the chat task below).

**Temporal filters** (03-api §Search, ADR-032): `since`/`until` = occurred-range window
(`COALESCE(occurred_end, occurred_start) >= since` / `occurred_start <= until`); `as_of` = simple
node-date `occurred_start <= as_of`. Applied to **all four sub-legs**; undated nodes fall outside any
window (literal-contract semantics). `SearchRequest` gains `since`/`until`/`as_of` (`date | None`),
threaded router → service → `RetrievalParams` (a value object bundling the RRF/recency knobs +
filters, so the store signature stays legible). Result shape (`SearchResultItem`) **unchanged**.

**Config** (Rule 9): `search_rrf_k` (60), `search_rrf_candidates` (60, clamped up to `top_k` so a
large `top_k` can't out-run the pool), `search_recency_half_life_days` (180, `Field(gt=0)` — SQL
divisor), `search_recency_floor` (0.9, `Field(ge=0, le=1)`).

**Verification:** `ruff` clean; **400 tests green** (+4 net: raw-query-unprefixed-to-FTS, candidate-
pool-≥-top_k both branches, temporal forwarding service+router; head-check 007→008). Hybrid SQL is
not fake-testable, so `scripts/smoke_db.py` gained **12 real-PG16 checks** (all green): FTS lexeme
match (`office`→{MEM1,MEM2}), non-English self-suppression → vector-only, and all three temporal
filters against seeded `occurred` dates. (Two **pre-existing, unrelated** smoke failures remain —
`list_entities`/`entities_touched_since` return real dev person nodes (Horia/Stefan) because the
local DB is now populated from the M3 prod reprocess; both checks predate this task and touch no
search code.)

**Independent review — APPROVE-WITH-MINORS, no must-fix** (fresh agent vs 08 §M4 kickoff-grill +
ADR-032/037 + 02/03/04 + CLAUDE rules + diff): confirmed rank-based RRF (no cosine/ts_rank blend),
per-leg candidate bounding, migration-008 spec + Rule-1 rebuildability, recency bounds, temporal
wording match on both legs, `websearch_to_tsquery` injection/crash safety, backward-safe `min_score`,
no ORM leak / config-only / result-shape-unchanged. **4 minors, 3 fixed:** div-by-zero guard on the
half-life (`Field(gt=0)` + floor bounds), UTC-pinned recency `today` (was `CURRENT_DATE`),
deterministic rank tiebreak (`, node_id`). **1 logged as a follow-up (MINOR-1):** the chat task's
"not in your memories" `min_score` floor (04-pipelines §5) must be tuned to the fused RRF×recency
scale (~0.03 max), **not** a cosine scale — a note for M4 task 3, not a defect here.

## Task 3

**Chat service — condense → hybrid retrieval → fenced grounded prompt → cited-only parse →
persistence; non-blocking `quick` titling** ([04-pipelines §5](../04-pipelines.md) +
[ADR-025](../adr/025-ui-editable-model-routing-and-per-task-effort.md)/[032](../adr/032-prior-art-adoptions.md)/[043](../adr/043-quick-routing-tier-m4.md);
03-api §Chat) — server, local, **not pushed**. The **service layer only** (routers = task 4).

**New `app/chat/` package.** `ChatService.send` runs one turn (`service.py`): resolve/implicit-create
the session → snapshot history (last `chat_condense_last_n`=15) → **persist the user turn BEFORE any
model call** (never-lose, rule 2) → **turn 1 uses the raw message, turn ≥2 CONDENSES to a standalone
ENGLISH query on the `conspect` group** (inherits conspect effort; degrades to the raw message when
the condense chain is down — `RegistryExhausted` subclasses `ProviderUnavailable`, and the English
corpus self-suppresses a raw non-English FTS query while vectors stay cross-lingual, ADR-032) →
**hybrid RRF retrieval** via the existing `SearchService` at the chat-tuned floor → **fenced**
grounded prompt → **cited-only `[n]` renumber** → persist the assistant turn (`model` = resolved
model incl. fallback, `sources` = cited nodes) + `set_last_model` → **best-effort, non-blocking
`quick`-tier titling** after the first exchange (background task, drained on shutdown).

- **`citations.py`** (pure): `renumber_citations` keeps only cited nodes and renumbers `[1..m]` by
  first appearance; out-of-range / `[0]` / too-big markers are **dropped, never an error**; repeats
  reuse their number; non-digit brackets untouched.
- **`prompts.py`**: grounded-answer prompt (grounding-biased, cite `[n]`, general questions uncited,
  "not in your memories" for personal questions with no hits, reply in the user's language),
  English-condense prompt, `quick` title prompt, and **context fencing** — each retrieved item is
  fenced as **data-not-instructions** (injection hygiene, before connector/MCP content shares the
  path — 04 §5).
- **`store.py`**: `PgChatStore` over `chat_sessions`/`chat_messages` (plain asyncpg, no ORM — rule 5);
  `sources` serialized + `::jsonb`-cast (mirrors the routing store); `session_messages(limit=…)` is a
  newest-N-then-reorder window (oldest-first).
- **MINOR-1 resolved (task 2 follow-up):** the chat "not in your memories" floor is a new config
  `chat_retrieval_min_score` (**0.01**) floored on the **fused RRF×recency** scale (~0.03 max: one leg
  at rank 1 ≈ 1/(k+1)·recency ≈ 0.015), **not** a cosine scale — `SearchService.search` gained a
  keyword-only `min_score` override (default `None` ⇒ the `/search` floor `search_min_score`, so the
  existing search path is byte-for-byte unchanged). New config: `chat_condense_last_n`,
  `chat_title_max_chars`. `main.py` wires `app.state.chat_service` (+ drain).

## Task 4

**Chat routers — `POST /chat`, `GET /chat/models`, `GET /chat/sessions[/{id}]`** (03-api §Chat,
[ADR-025](../adr/025-ui-editable-model-routing-and-per-task-effort.md)/[043](../adr/043-quick-routing-tier-m4.md))
— server, local, **not pushed**. A thin HTTP surface over the task-3 `ChatService` (rule 5:
validate + delegate; no logic in the router).

**New `app/routers/chat.py`** (no prefix, `tags=["chat"]`, session-gated like every non-public route),
wired into `main.py`, reached via a new `get_chat_service` dependency:
- **`POST /chat`** → `ChatService.send` → `ChatResponse` (`session_id`, `answer`, `model_used`,
  `fallback_used`, cited `sources`). Error mapping (the task's explicit requirement):
  **`RegistryExhausted` → 503** (whole chat chain down — the user turn is already durably persisted,
  never-lose rule 2) and **`ChatSessionNotFound` → 404**. The composer's `model` picker + `planes`/
  `top_k` are forwarded; `fallback_used`/`model_used` surfaced, never swallowed (rule 3).
- **`GET /chat/models`** → `{ models: [{id, label}], default }` via a new `ModelRoutingService.
  chat_catalog()` over `ProviderRegistry.chat_models()`. `default` = the **Chat group's active model**
  (`resolve("chat").chain[0]`, saved-over-seed). Lists every **genuinely** chat-capable provider.
- **`GET /chat/sessions`** → `[{id, title, created_at, last_model}]` newest-first (bounded by a new
  `chat_sessions_list_limit` config, rule 9); **`GET /chat/sessions/{id}`** → `{id, title, messages:
  [{role, content, model, sources, created_at}]}`, 404 on unknown. Read paths are new
  `ChatService.list_sessions` / `get_session_detail` methods (service, not store, per rule 5).
- **New wire models** in `models.py` (`ChatRequest`/`ChatResponse`/`ChatSourceItem`/`ChatModelItem`/
  `ChatModelsResponse`/`ChatSessionItem`/`ChatMessageItem`/`ChatSessionDetail`).

**Provider `can_chat` + `label` seam** (small, provider-sourced like `supports_effort`): the OpenAI-
compatible class also backs STT/embedding (ADR-004), so it's a `ChatProvider` by *class* even with no
chat model configured. Added `ChatProvider.can_chat` (default True; `OpenAICompatibleProvider` sets it
`bool(default_chat_model)`) so `chat_models()` filters on real capability, and a provider-sourced
`label` (the configured model; falls back to the id).

**Session ids validated as uuids at the boundary:** `chat_sessions.id` is a uuid column, so a malformed
id would 500 in the store. Both entry points now reject it early — `ChatRequest.session_id: UUID | None`
(422 on a bad body id) and the `GET` path typed `uuid.UUID` (422, consistent with `GET /nodes/{id}`) —
so the str-based store contract only ever sees valid uuids; a well-formed-but-absent id is a clean 404.

**Verification:** `ruff` clean; **439 tests green** (+16: 12 in `test_chat_router.py` — renumbered-citation
answer + implicit session, picker/filter forward, existing-session continuation, unknown→404, malformed→422,
all-models-down→503 with the user turn still persisted, empty-message→422, models catalog ids/labels/default
with a non-chat provider excluded, sessions newest-first, detail with persisted-source round-trip; +4 read-path
+ catalog cases in `test_chat_service.py`). `FakeChatStore` now mints real uuid session ids (it had diverged
from the uuid column — the source of two masked bugs).

**Independent review — APPROVE after fixes** (fresh agent vs 03-api §Chat + 08 M4 kickoff + ADR-025/043 +
CLAUDE rules + diff). Contract match, layering, rule-3/9 all confirmed clean. **2 must-fix, both resolved
+ regression-tested:** (1) `chat_models()` filtered on the `ChatProvider` *class*, so the real registry

change needed — the implementation conforms to 03-api §Chat as written.

## Task 5

**Settings routers — `GET /settings` + `PUT /settings/models`** (03-api §Settings,
[ADR-025](../adr/025-ui-editable-model-routing-and-per-task-effort.md) §6 / [ADR-043](../adr/043-quick-routing-tier-m4.md))
— server, local, **not pushed**. The Settings → Models panel surface over `ModelRoutingService` (rule 5:
validate + delegate). The router previously held only the M3 `PUT /settings/vocabulary` seam.

- **`GET /settings`** → `{ groups: [{group, active, fallback, effort_by_provider, models}] }` for **all
  3 groups** (`chat`/`conspect`/`quick`, in `GROUPS` order). Each group's routing is the **effective
  saved-over-seed** `GroupRouting`; `models` is the registry's pickable list, each `{id, label,
  supports_effort, effort_levels}` — so the web renders the effort selector only where it applies, from
  **registry-sourced** levels (ADR-025 §6, no hardcoded model/effort lists in web).
- **`PUT /settings/models`** → saves **one** group (`ModelRoutingUpdate{group, active, fallback,
  effort_by_provider}`, `group` a `Literal` of the 3 tiers → 422 on a bad group) and busts the routing
  cache (**forward-live**, `save()`→`invalidate()`). Validated in `ModelRoutingService.save_group`
  against `registry.chat_models()`: unknown active/fallback id, effort on a non-effort model, or an
  invalid effort level all raise **`UnknownModel` → 422** *before* any write (nothing bad reaches
  `app_settings`). Returns the updated group's editable state.
- **Service:** new `GroupSettings` dataclass, `all_settings()` (per-group effective routing + models),
  validating `save_group()`, and `UnknownModel`. **Effort levels are provider-sourced:** new
  `ChatProvider.effort_levels` (`claude_max` = `low|medium|high|xhigh|max`), surfaced through an extended
  `ChatModelOption` (`supports_effort`/`effort_levels`, defaulted so `GET /chat/models`'s id/label mapping
  is unchanged).

**Label mechanism (ADR-025 §6 note):** the ADR sketched a `CHAT_MODEL_LABELS` config map; the build
instead sources `label` from the provider's configured model (the task-4 convention). The binding outcome
— registry-sourced labels, no hardcoded web lists — holds; reviewer confirmed this is an acceptable
equivalent, not a contract violation. No contract-doc change needed.

**Verification:** `ruff` clean; **449 tests green** (+12 in `test_settings_models_router.py`: all 3 groups
from seed, models carry effort capability/levels + non-chat provider excluded, PUT saves + **forward-live
cache-bust verified end-to-end**, a non-`chat` group saves, and the full 422 matrix — unknown active/
fallback/group, effort on a non-effort model, invalid level, missing active). `FakeChatProvider` gained
`effort_levels`/`label`/`can_chat` params.

**Independent review — APPROVE, no must-fix** (fresh agent vs 03-api §Settings + ADR-025 §6/§2/§3 + ADR-043
+ CLAUDE rules + diff): contract match (all 3 groups, registry-sourced models/levels), rule-5 layering,
forward-live cache-bust, the full pre-write 422 validation, task-4 consistency (extended `ChatModelOption`
didn't change `GET /chat/models`), and the label deviation all confirmed clean. **2 minors, both addressed
in-session:** the flagged test edges (non-chat group save, unknown-fallback 422, PUT-response `models`
assertion) were **added**; the other — `all_settings` displays saved routing verbatim while `resolve()`

staleness kept deliberately (an *editor* should show exactly what the user saved, not a silently-degraded
view), logged as a follow-up should the UI ever need the run-time-effective routing.

## Task 6

**Web chat screen** (06-web-app §2 / [ADR-025](../adr/025-ui-editable-model-routing-and-per-task-effort.md) /
[ADR-043](../adr/043-quick-routing-tier-m4.md)) — web, local, **not pushed**. The PWA's Chat tab, a thin
TanStack-Query client over the task-4/5 API (03-api §Chat); no server code touched (ADR-006). Replaces the
`ComingSoon` placeholder.

**Wire seam** (`api/types.ts` + `api/client.ts`): the `Chat*` types (`ChatSourceItem`/`ChatRequest`/
`ChatResponse`/`ChatModelItem`/`ChatModelsResponse`/`ChatSessionItem`/`ChatMessageItem`/`ChatSessionDetail`)
+ `chat`/`chatModels`/`listChatSessions`/`getChatSession` methods — cross-checked field-for-field against the

**`features/chat/useChat.ts`:** `useChatModels` (picker catalog, 60s stale — the Settings Chat-group default

untitled session sits inside a 90s window, so the best-effort `quick`-tier title lands without polling
forever), `useChatSession` (lazy detail on open), `useSendChat` (mutation; invalidates only the **list** key
so optimistic turns survive).

**`features/chat/ChatScreen.tsx`:** implicit-session chat loop over local thread state.
- **List / open / new** only (rename+delete deferred — no endpoints). A collapsible History panel; opening a
  session **seeds the thread once** from `GET /chat/sessions/{id}`, guarded by a `seededFor` ref so a send's

  review traced new/existing/re-open/A→B→A/focus-refetch — race-free).
- **Composer:** per-conversation **model picker** (`GET /chat/models`, default = Chat-group active; resets to
  default on new/open so an override never leaks across threads), **plane-filter chips** (reuses the shared
  `usePlanes` meta read → `POST /chat` `planes`), auto-growing textarea, **Enter-to-send** (Shift+Enter =
  newline, IME-safe).
- **Client-side reveal:** the non-streaming answer is tokenized into words + inline `[n]` badges and faded in

## M4 follow-up 3 · Task 2 — migration + legacy-tolerant labels (2026-07-15)

**Built** (commits `6133296`/`74de20d`, local, **not pushed**): the P10-load-bearing step of ADR-045 §4 —
migrate the saved routing config in place so a deliberate choice survives the provider→model rename instead of
degrading to the seed (the exact deploy-ordering gap task 1 flagged).

- **migration 009** (`009_model_routing_id_migration.py`, `down_revision="008"`) — an **idempotent, plain-SQL**
  (ADR-011) data migration over the one `app_settings.model_routing` jsonb row. Rewrites, in `active`/`fallback`
  **and the effort object's keys**: `claude-max`→`claude-opus-4-8`, `claude-max-sonnet`→`claude-sonnet-4-6`,
  `nebius`→`meta-llama/Llama-3.3-70B-Instruct`, **plus renames the key `effort_by_provider`→`effort_by_model`**
  (task 1 renamed it in code with no migration, so pre-existing saved data would otherwise lose its effort). The
  key rename is required to make "saved routing survives the rename" actually hold — recorded so it's not a
  surprise vs the ADR bullet, which lists only the value remaps.
- **mechanism** — an **ordered text substitution** over the jsonb serialization (`value::text` → `replace(...)` ×4
  → `::jsonb`). The value domain is a *closed* set (model/provider ids, the effort-key name, `low`/`medium`/`high`,
  group names), so no search token is a substring of any legitimate value we don't rewrite. Ordering matters only
  for the one prefix hazard (`claude-max` ⊂ `claude-max-sonnet`): the `-sonnet` form is replaced **first**. No
  replacement's output contains a later token → single-application, unambiguous. (A structured `jsonb_object_agg`
  key-remap was considered; it must rebuild each group object and guard the empty-row NULL-collapse — more surface
  for the same closed-set result.)
- **idempotent + no-op** (rule 6) — a `WHERE key='model_routing' AND value::text ~ '<old-tokens>'` guard skips any
  absent / empty `{}` / already-migrated row (no spurious write, no NULL-collapse), and a re-run matches nothing
  (all old tokens gone after pass 1). `downgrade()` reverses the map (best-effort, ADR-011) and round-trips.
- **legacy-tolerant labels** (ADR-045 §6) — `friendly_model_label` now **folds** the retired provider ids
  (`claude-max`/`claude-max-sonnet`/`nebius`) to their vendor string *first*, then derives the label the normal
  way → "Claude Opus 4.8" / "Claude Sonnet 4.6" / "Llama 3.3 70B". So historical `chat_messages.model` rows —
  **left untouched** (rewriting audit would falsify a turn recorded during the broken-Llama-3.1 window, ADR-045
  §4) — still render a friendly name. The `_LEGACY_MODEL_IDS` map is a retired-id compat shim kept in lock-step
  with migration 009's three pairs (rule 9 intact — live models are still config-declared). *Web consumption of
  the label for history is task 4; the migration itself never touches `chat_messages`.*
