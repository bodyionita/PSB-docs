## Task 1 — Server (feed filter, recursive run tree, all-source captures)

**The server half of M8.1** (ADR-054 §2/§4/§5). Solo task, first in the milestone — Task 2
(web primitives) then Batch C {T3 Explore, T4 Activity/Captures} build the web against it.
**No migration.** Server-only. **Committed locally, not pushed.**

### What was built

- **Feed = parentless runs only (ADR-054 §2).** `activity_feed.py` `_AGENT_RUNS_BRANCH` now filters
  `WHERE r.parent_run_id IS NULL` (the trigger predicate is ANDed on when only one of the two run
  categories is requested; both → the `CASE` assigns each row by origin). A pipeline run is one feed
  row; its step children live under the run-detail tree. `parent_ref` stays on the wire (always null
  on that branch now). The feed gets dramatically quieter (a 12-step night collapses to 1 row).
- **Recursive `children[]` on `GET /activity/runs/{id}` (ADR-054 §2).** New in `agent_runs.py`: a
  `RunChild` node (`{id, name, status, ts, summary, children[]}`), a `RunTreeRow` flat DTO, a **pure
  `build_run_tree(rows, root_id)`** that folds the flat descendant list into the forest (siblings
  early→late; every node pre-created so attachment is order-independent; an orphan whose parent isn't
  in the set is kept at top level, never dropped — rule 7), and `PgAgentRunStore.children_tree` = a
  **recursive CTE** over `parent_run_id` bounded by `RUN_TREE_MAX_DEPTH=32` (cycle insurance; the
  linkage is a DAG by construction), `ORDER BY started_at ASC NULLS FIRST, id ASC`. `AgentRunResponse`
  gains a recursive `RunChildModel children[]`; the router fetches the tree only on the drill-down.
  - **Depth note (ADR-050 interaction).** A step's spawned `capture` runs parent to the pipeline
    *root* (not the step), so current real trees are **one level deep** — steps and spawned runs are
    siblings under the parent. The render is genuinely recursive (proved by a depth-2 unit + smoke
    case), so any future deeper nesting shows its true depth; today it's flat by design.
- **Captures branch → all sources + `status`/`source` (ADR-054 §4).** `_CAPTURES_BRANCH` dropped the
  `source = 'chat'` filter (keeps `removed_at IS NULL`); the **`conversations` category is renamed
  `captures`** and the normalized `kind` is renamed `chat_capture`→`capture` (the branch is no longer
  chat-only). Every feed row now carries `status` (source-row lifecycle) and `source` — for a capture
  `COALESCE(source, kind)` → `text`/`voice`/`mcp`/`chat`, null on the non-capture branches. The
  **expand detail fetch** is the existing `GET /captures/{id}`; its `CaptureView` gained `source` so
  the expanded card renders the badge self-contained. `ref` keeps the chat-session id (`source_ref`);
  the client expands via the row `id`.
- **graph-health node ids (ADR-054 §5) — already satisfied, no code change.** `graph_health.py`'s
  `Offender` has emitted `{id, label}` since M8 Batch B T4, so `details.checks[].sample` already
  carries ids. **Carve-out recorded** (03-api §Activity): the six *node*-checks carry a `nodes.id`
  (web → clickable `NodeChip`); the **`pending-review-aging`** check's offenders carry a
  `review_queue.id` — a review item, not a single node (stance-candidate/dedup items intrinsically
  have no one node id) — so the web links those to the Review item, not `NodePreview`. This **refines
  ADR-054 §5's** "all offender samples carry node ids" (which was accurate only for the node-checks).

## Replan — T2 scope (2026-07-17, grilled, refines ADR-054 §5)

A T2 implementation session opened, read the docs, and immediately hit a contract-vs-data conflict
that the docs didn't resolve, so it **stopped and replanned** (09 §Implementation-session loop rule 4)
and grilled the scope decision-by-decision with the user. No code was written. Outcome below; the
authoritative task text is [../08-implementation-plan.md](../08-implementation-plan.md) §M8.1 T2/T4.

**The conflict.** ADR-054 §5 lists **"capture node chips"** among the references `NodeChip` should make
clickable → `NodePreview`. But `NodePreview` fetches a node by **frontmatter uuid** (`GET /nodes/{id}`
takes a `uuid.UUID`), while a capture's resulting nodes are exposed only as **store paths**
(`CaptureView.node_paths: string[]`) — and per [../02-data-model.md](../02-data-model.md) §Identity the
uuid is identity, **"paths are projections."** The capture record stores paths, not ids (confirmed
server-side). So capture chips **cannot** open a preview web-only; it needs a server change. T1 did not
add that field, and smuggling a lone server change into a "web primitives" task is wrong.

**Decisions (grilled, decision-by-decision):**
1. **Scope/sequencing → defer capture chips to T4.** T2 ships the reusable primitives and wires only
   uuid-bearing surfaces; capture/Captures chip clickability + the server node-id exposure land in **T4**
   (which already builds the Captures surface and will make the same server change — a read-time
   `node_paths → nodes.id` join, **no migration**). Refines ADR-054 §5, same mechanism as T1's carve-out.
2. **Drawer mechanism → one app-level `NodePreview` bottom-sheet drawer** via a new `useNodePreview()`
   context (mirrors `MapNavContext`, mounted in `AppShell`; T2 is sequential *before* batch C, so T3's
   later 7→6 `AppShell` rebuild sits on top). A tiny chip inside a flex-wrap row can't expand a
   full-width inline preview — a global drawer is the only thing that makes "clickable anywhere" uniform.
3. **"Explore in map" lives in the drawer chrome; `NodePreview` is unchanged** (Search already renders
   its own map button + passes `onOpenNode` for edges — adding a root button into `NodePreview` would
   double it). The "second hop" is a property of the drawer, not the preview body.
4. **Drawer edge chips → jump to Map (close drawer), identical to Search/Chat** (rejected: in-drawer
   node→node re-centering — it would diverge `NodePreview` per-consumer and build a drawer history;
   ADR-054 §5's "click → card → map" says the card's forward motion *is* into the map).
5. **`pending-review-aging` offenders → a new Review deep-link** (user chose to build it over leaving
   them static): a `ReviewNavContext.openReviewItem(id)` from `AppShell` → Review tab, **scroll-to +
   transient reduced-motion-safe highlight**, searching **pending ∪ maybe** (the check spans both
   `DECIDABLE_STATUSES`); a **stale/resolved** id (gone since the nightly snapshot) → **silent land** on
   the tab. Rendered as a **separate review-chip** — `NodeChip` stays strictly node-uuid-only.
6. **`<TimeAgo>`** wraps `relativeTime` **unchanged** (coarse "just now / Nm / Nh / Nd ago", even "400d
   ago" — user-confirmed this is the desired display) + a **custom** tap(mobile)+hover(desktop) tooltip
   with the exact local time (`17 Jul 2026, 08:36`, 24h, no seconds; also `aria-label`); reduced-motion
   fade. Swap **all 9** `relativeTime` sites **+** SettingsScreen's absolute `session_created_at` stamp.
7. **Minor/props:** bottom-sheet drawer (slide+backdrop; dismiss via backdrop/Esc/close; reduced-motion
   → fade); `openNode(id, hint?: {type, title})` for an instant header; `<NodeChip nodeId type title />`
   null-safe pill; graph-health node-check offenders adopt `NodeChip`, replacing the static pills.

**Recording:** no edit to ADR-054 (ADRs are never edited) and **no new ADR** — these are milestone-plan
refinements + a minor UI-nav affordance (the Review deep-link), well within ADR-054's "universal
clickable references" intent. Recorded in 08 §M8.1 T2/T4 + this note + a README status line. **Paused
before implementation** per [../09-session-protocol.md](../09-session-protocol.md).

## Task 2 — Web primitives (`<TimeAgo>` + `NodeChip`/`NodePreview` drawer + Review deep-link)

Built to the replan above (ADR-054 §1/§5). **Web-only — no server code touched (ADR-006).** Solo task,
before Batch C. **Committed locally, not pushed** (`20a8fba` + review-fix `ee229f0`).

### What was built

- **`<TimeAgo>`** (`ui/TimeAgo.tsx`) — wraps the shared `relativeTime` **byte-for-byte unchanged**
  (coarse "just now / Nm / Nh / Nd ago", even "400d ago") and adds a **custom tooltip** showing the
  exact local time (`exactTime()`, new sibling in `relativeTime.ts` — `17 Jul 2026, 08:36`, 24h, no
  seconds, local-getter-built so it's locale-order-stable). Tooltip on **hover (desktop) + tap
  (mobile)**: hover is gated to a **mouse pointer** (`onPointerEnter/Leave`, `pointerType==='mouse'`)
  so a touch tap's synthetic `pointerenter→click` can't open-then-close on the first tap; touch/pen
  taps use the click toggle. Dismiss on outside-tap / scroll / Esc; the exact time (+ the relative
  phrase) rides in `aria-label`; reduced-motion-safe fade; `stopPropagation` so tapping a timestamp
  inside a clickable row doesn't trigger the row. **All 9 `relativeTime` call sites** swapped
  (AutoRecordedList, ProvidersPanel ×2, RecentCaptures, ReviewScreen's aging tag, FeedView, GraphHealth,
  OpsView ×2) **+ SettingsScreen's** absolute `session_created_at` stamp → "Session started …".
- **`NodeChip` → one app-level `NodePreview` bottom-sheet drawer.** `ui/NodeChip.tsx` (type icon +
  title pill, **strictly node-uuid-only**; degrades to a static pill with no provider) calls a new
  `useNodePreview()` context (`ui/nodePreviewNav.ts`, mirrors `mapNav`) whose single drawer
  (`ui/NodePreviewDrawer.tsx`) is mounted in `AppShell`. The drawer chrome owns the header (icon +
  title from the chip's `hint` for an instant paint + Close) **and** the "Explore in map" hop;
  **`NodePreview` is unchanged** — its edge chips still take `onOpenNode` and jump to the Map, uniform
  across Search/Chat/drawer. `onExploreInMap` closes the drawer **and** navigates. Sheet entrance is a
  **pixel rise** (`y: 40 → 0`, not `y: '100%'` — a percentage transform can stick at its height-derived
  initial before the sheet is measured, leaving it off-screen); reduced-motion → fade.
- **Graph-health offenders wired (`GraphHealthCard.tsx`).** The six **node-checks** (offender id =
  `nodes.id`) render as **`NodeChip`**; **`pending-review-aging`** (offender id = a `review_queue.id`)
  renders as a distinct **review-chip** → a new **Review deep-link** (`features/review/reviewNav.ts`,
  `ReviewNavContext.openReviewItem` from `AppShell` → Review tab). `ReviewScreen` gained a `seed` prop:
  once both lists load it finds the item (**pending ∪ maybe**), expands Parked if needed,
  **scroll-into-view + a transient reduced-motion-safe ring** (auto-clears ~2.8s); a stale/resolved id
  → **silent land**. The seed is **not consumed in the child effect** (React 18 StrictMode double-mounts
  the screen — consuming during the throwaway first mount would blank the real mount's highlight);
  `AppShell` instead clears `reviewSeed` on **manual bottom-nav** navigation, and reaching the offender
  chip always remounts Review, so a stale re-highlight is unreachable. `NodeChip` stays node-only.

### Verification

- tsc + eslint (`--max-warnings 0`) + vite build **green**.
- **Real-browser walkthrough vs a throwaway mock API** (auth/me + agents/pipelines + graph-health run
  with node + review offenders + nodes/{id} + review lists). Drove: graph-health **`NodeChip` → drawer**
  (header, live body, canonical + derived edges) → **"Explore in map"** closes the drawer and lands on
  the **Map centered on the node**; the **review-chip deep-link** → **Review tab + accent ring on the
  target, auto-cleared by ~2.8s**; **`<TimeAgo>`** exact-time `aria-label` + click-to-open tooltip.
  *(The preview browser tab is `document.hidden`, which freezes framer-motion paint/animation
  app-wide — an environment artifact, not a bug; behavior was confirmed via DOM/state inspection. The
  iOS-Safari touch-tap path is verified-by-construction + gets real-device confirmation at T5.)*
- Verification itself **caught 3 real bugs, all fixed before review:** a deep-link timer-teardown
  (effect cleanup cancelled the scroll/clear timers), the **StrictMode double-mount** highlight blank,
  and the drawer's percentage-`y` stuck-off-screen entrance.

### Independent review — APPROVE-WITH-MINORS (1 must-fix + 2 minors applied, `ee229f0`)

A fresh independent reviewer re-derived scope from ADR-054 + the replan + 08 + CLAUDE.md and traced the
diff. Verified clean: ADR-006 (zero server code / no cross-import), all 7 replan decisions, provider
nesting + closing tags, the deep-link's StrictMode + warm-cache + one-shot-timer robustness,
reduced-motion coverage, no debug/dead code.

- **MUST-FIX (resolved):** `<TimeAgo>` used `onMouseEnter/Leave` — on **iOS Safari** the synthetic
  `mouseenter→click` on the first tap opens-then-closes, showing nothing (defeats ADR-054 §1's whole
  reason for a custom tooltip). Fixed → pointer-gated to a mouse pointer; touch uses the click toggle.
- **Minors (applied):** `aria-label` now carries the relative phrase **+** exact time (was exact only);
  drawer `onClose` memoized (`useCallback`) so its keydown effect stops re-subscribing every render.
- **Follow-up (logged, out of scope):** `NodePreviewDrawer` declares `role="dialog"`/`aria-modal` but
  has no focus trap/restore — low priority for a single-user app.
