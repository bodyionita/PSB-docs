## Task 1 — Server (feed filter, recursive run tree, all-source captures)

**The server half of M8.1** (ADR-054 §2/§4/§5). Solo task, first in the milestone — Task 2
(web primitives) then Batch C {T3 Explore, T4 Activity/Captures} build the web against it.
**No migration.** Server-only. **Committed locally, not pushed.**

### What was built

- **Feed = parentless runs only (ADR-054 §2).** `activity_feed.py` `_AGENT_RUNS_BRANCH` now filters
  `WHERE r.parent_run_id IS NULL` (the trigger predicate is ANDed on when only one of the two run
  categories is requested; both → the `CASE` assigns each row by origin). A pipeline run is one feed
  row; its step children live under the run-detail tree. `parent_ref` stays on the wire (always null
  on that branch now). The feed gets dramatically quieter (a 12-step night collapses to 1 row).
- **Recursive `children[]` on `GET /activity/runs/{id}` (ADR-054 §2).** New in `agent_runs.py`: a
  `RunChild` node (`{id, name, status, ts, summary, children[]}`), a `RunTreeRow` flat DTO, a **pure
  `build_run_tree(rows, root_id)`** that folds the flat descendant list into the forest (siblings
  early→late; every node pre-created so attachment is order-independent; an orphan whose parent isn't
  in the set is kept at top level, never dropped — rule 7), and `PgAgentRunStore.children_tree` = a
  **recursive CTE** over `parent_run_id` bounded by `RUN_TREE_MAX_DEPTH=32` (cycle insurance; the
  linkage is a DAG by construction), `ORDER BY started_at ASC NULLS FIRST, id ASC`. `AgentRunResponse`
  gains a recursive `RunChildModel children[]`; the router fetches the tree only on the drill-down.
  - **Depth note (ADR-050 interaction).** A step's spawned `capture` runs parent to the pipeline
    *root* (not the step), so current real trees are **one level deep** — steps and spawned runs are
    siblings under the parent. The render is genuinely recursive (proved by a depth-2 unit + smoke
    case), so any future deeper nesting shows its true depth; today it's flat by design.
- **Captures branch → all sources + `status`/`source` (ADR-054 §4).** `_CAPTURES_BRANCH` dropped the
  `source = 'chat'` filter (keeps `removed_at IS NULL`); the **`conversations` category is renamed
  `captures`** and the normalized `kind` is renamed `chat_capture`→`capture` (the branch is no longer
  chat-only). Every feed row now carries `status` (source-row lifecycle) and `source` — for a capture
  `COALESCE(source, kind)` → `text`/`voice`/`mcp`/`chat`, null on the non-capture branches. The
  **expand detail fetch** is the existing `GET /captures/{id}`; its `CaptureView` gained `source` so
  the expanded card renders the badge self-contained. `ref` keeps the chat-session id (`source_ref`);
  the client expands via the row `id`.
- **graph-health node ids (ADR-054 §5) — already satisfied, no code change.** `graph_health.py`'s
  `Offender` has emitted `{id, label}` since M8 Batch B T4, so `details.checks[].sample` already
  carries ids. **Carve-out recorded** (03-api §Activity): the six *node*-checks carry a `nodes.id`
  (web → clickable `NodeChip`); the **`pending-review-aging`** check's offenders carry a
  `review_queue.id` — a review item, not a single node (stance-candidate/dedup items intrinsically
  have no one node id) — so the web links those to the Review item, not `NodePreview`. This **refines
  ADR-054 §5's** "all offender samples carry node ids" (which was accurate only for the node-checks).

### Verification

- **816 unit tests** (+7: 5 `build_run_tree` in the new `test_run_tree.py` — flat/nested/orphan/
  out-of-order/empty; 2 router tests — leaf empty `children`, a depth-2 recursive tree rendered
  early→late), ruff check + format clean.
- **Real-PG smoke 149/149** (+13): `check_run_children_tree` (parent→two steps→a depth-2 grandchild;
  ordering; leaf/unknown → empty forest) and `check_activity_feed_m81` (parentless-only filter; chat
  vs web-text captures with `source`/`status`; `COALESCE(source, kind)` fallback; `removed_at`
  exclusion; category narrowing). These exercise the un-fakeable SQL (the recursive CTE, the parent
  filter, the coalesce) — the fake-backed unit tests can't.

### Independent review — APPROVE after fix (1 must-fix, resolved)

A fresh independent reviewer read ADR-054 + 03-api + ADR-050 + CLAUDE.md invariants + the diff.
Verified clean: the recursive CTE (depth-bound, ordering, casts), `build_run_tree` (orphan/order
handling), UNION column parity + injection-safety, the `conversations→captures` / `chat_capture→
capture` renames as faithful-to-ADR-054-§4 (not an unrecorded decision), the recursive
`RunChildModel` serialization + no circular import, `removed_at` exclusion.

- **MUST-FIX (resolved):** the 03-api addendum I first wrote overclaimed that *all* graph-health
  offender samples are node ids — false for `pending-review-aging` (a `review_queue.id`). The **code
  was correct** (it emits the right id per check); the **contract wording was wrong**. Fixed by
  recording the accurate carve-out (above) in 03-api §Activity — node-checks → `NodeChip`, review-
  aging → Review item — refining ADR-054 §5. No code change (a review item has no single node id to
  emit; forcing one would be wrong).
- **Minors (logged):** captures-branch `ref` is chat-session-or-null (drill-down keys off row `id`,
  documented in the branch comment + addendum); the SQL-only paths are covered by the smoke, not the
  fake unit tests (expected per rule 5 / testing policy — smoke run confirmed green).
