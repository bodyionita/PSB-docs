## Task 1

**`chat_distill_state` migration + the chat-distiller job** ([ADR-048](../adr/048-m6-chat-distiller-build-decisions.md)).
The stance-gated distiller that turns idle in-app chat sessions into memories ‚Äî "the sleep cycle"
for chat. **Not yet scheduled** (it becomes a `nightly` pipeline step in task 8); a standalone
`python -m app.cli chat-distill` verb is the run-now + local-test path. **Committed locally, not
pushed.**

### What was built

- **Migration 013** (plain SQL, ADR-011): the **`chat_distill_state`** watermark table (`session_id`
  pk ‚Üí `chat_sessions` `ON DELETE CASCADE`, `last_message_at`, `distilled_at`, nullable `run_id`) +
  a nullable **`captures.source_ref`** column. ADR-048 ¬ß1 mandates the endorsed capture carry
  `source_ref = <session-id>`, but the table had no such column and 02-data-model's captures
  paragraph didn't list one ‚Äî a doc/ADR gap. Resolved by adding the column (mirrors `nodes.source_ref`)
  and aligning [02-data-model](../02-data-model.md) (not a replan: the ADR already *decided* it; only
  the mechanic was unstated). Additive + reversible (up/down/up round-trip verified on local pg).
- **`PgChatDistillStore`** ([app/chat/distill_store.py]) ‚Äî the idle-eligibility query
  (`GROUP BY`/`HAVING max(created_at) < cutoff AND (watermark IS NULL OR max > watermark)` over a
  `LEFT JOIN`), **oldest-first** delta-after-watermark, and the watermark upsert.
- **`ChatDistillerService`** ([app/chat/distiller.py]) ‚Äî one `conspect` pass over a session's
  fenced delta returns `{candidate_text, stance, salience, evidence_excerpt, referenced_entity_names,
  why_unclear}` candidates; the stance gate routes **endorsed ‚Üí a `captures` row** (`source=chat`,
  `source_ref=session-id`, `created_at` = the anchoring message's time, via the capture pipeline =
  the single writer, rule 2b) ‚Üí organizer; **unclear ‚Üí a `stance-candidate` review item** (names +
  text, never node ids ‚Äî ADR-048 ¬ß7); **rejected ‚Üí run-log only**. Unknown stance biases to
  `unclear` (never guessed to endorsed); a light **hedge post-check** downgrades hedged "endorsed"
  to review; within-run **content dedup** guard; per-session best-effort; every run in `agent_runs`.
- **`CapturePipeline.create_chat_capture`** ‚Äî the endorsement‚Üíorganizer bridge, with a
  **deterministic capture id** (uuid5 over session + normalized statement) so a re-distill collapses
  instead of duplicating (rule 6). Extracted a shared **`build_capture_pipeline`** factory (rule 10)
  now used by both `reprocess.py` and the distiller CLI. Generalized the MCP burst semaphore to a
  `_organize_burst` shared by MCP + chat.
- **`KIND_STANCE_CANDIDATE`/`KIND_DEDUP_PROPOSAL`** constants; config knobs (`chat_distill_idle_hours`
  = 12h, `_max_sessions_per_run`, `_max_delta_messages`, `_max_candidates`); the `chat-distill` CLI verb.

### Independent review ‚Äî APPROVE after fixes (2 must-fix, both resolved)

A fresh reviewer read ADR-048 + the docs + the full diff. Confirmed contract fidelity (one captures
row/endorsed candidate, exact ¬ß7 payload, rejected‚Üílog-only, stance never guessed, run never crashes,
migration additive/reversible) and cleared the `captures.source_ref` decision + the `build_capture_pipeline`
refactor. Two **must-fix** idempotency/visibility findings, both fixed + regression-tested:

1. **Cross-run duplicate materialization** ‚Äî a partial per-session failure (one candidate's capture
   write fails after earlier ones succeeded) left the watermark un-advanced ‚Üí the next run re-emitted
   the already-written candidates as duplicate nodes. **Fixed** by the deterministic capture id +
   already-materialized skip in `create_chat_capture` (rule 6: retries are safe).
2. **Silent delta truncation** ‚Äî an oversized delta kept the *newest* N and advanced the watermark
   past the *older* skipped messages, permanently stranding them (and the config promised
   "dropped-and-logged"). **Fixed** by switching the delta to **oldest-first bounded batches**,
   advancing the watermark only to the last message actually processed (so the remainder is deferred
   to the next run, and the run records `truncated`) ‚Äî a deferral, never a silent skip.

Minors logged as follow-ups (accepted, not blocking): node frontmatter carries `source=chat` but not
`source_ref` (the capture row does ‚Äî audit chain is node‚Üícapture‚Üísession); equal-timestamp watermark
tie (rare at ¬µs resolution; strict `>` kept for idempotency); `_anchor_time` prefix match can pick a
wrong message (impact = the memory's timestamp only; sane fallback).

### Verification

- **639 unit tests** green (+20: distiller parse/fence/hedge/anchor + stance-gate + truncation +
  idempotency + capture-source), **ruff clean**.
- **Migration 013** up/down/up round-trip verified on local pg16.
- **Real-PG smoke** 86/86 (+7): the un-fakeable eligibility `GROUP BY`/`LEFT JOIN`, oldest-first
  delta, watermark upsert-in-place + re-eligibility, and `captures.source_ref` round-trip.
- **CLI end-to-end on real dev data + real LLM**: `chat-distill` distilled 6 idle sessions ‚Üí 1
  endorsed ‚Üí a `captures` row (`source=chat`, `source_ref=<session-id>`, node **dated to the
  conversation day** not the run time ‚Äî anchor-time works) ‚Üí organizer ‚Üí memory + event nodes ‚Üí
  indexed; a second run = **0/0 sessions** (delta-after-watermark idempotency). *(This mutated the
  local dev graph store: one real chat-distilled node + 6 session watermarks ‚Äî reversible via git;
  not run against prod.)*

## Task 2

**review_queue M6 kinds** ([ADR-048](../adr/048-m6-chat-distiller-build-decisions.md) ¬ß7): the
maybe-reopen fix, the `stance-candidate` agree=auto-endorse path, and the kind-aware reprocess
reset. Service/store only ‚Äî the web Review surface for these is task 7. **Committed locally, not
pushed.**

### What was built

- **Maybe-reopen (the load-bearing guard fix).** `PgReviewQueue.resolve` guarded `WHERE
  status='pending'`, which wrongly made `maybe` terminal. Now the guard is **decidable = `pending`
  ‚à™ `maybe`** (`WHERE status = ANY($::text[])`, a new `DECIDABLE_STATUSES` constant); `resolved`/
  `discarded` stay terminal. `ReviewService.resolve` mirrors it (`record.status not in
  DECIDABLE_STATUSES` ‚Üí `ReviewNotPending`/409) and so does `FakeReviewQueue`. This is generic ‚Äî an
  entity-ambiguity `maybe` is now re-openable too, matching the 02-data-model wording.
- **`stance-candidate` resolution** (`ReviewService._resolve_stance_candidate`, verdict
  `agree|disagree|maybe`): **agree ‚Üí the exact auto-endorse path** ‚Äî it calls the *same*
  `CapturePipeline.create_chat_capture` the distiller's endorsed branch uses (one ingest path, not
  two ‚Äî rule 10 / rule 2b), passing only the candidate's clean prose as the capture raw (organizer
  re-resolves everything, ADR-048 ¬ß2), idempotent via the deterministic capture id (rule 6);
  **disagree ‚Üí discarded** (logged, never a node); **maybe ‚Üí parked**, re-openable. The pipeline is
  injected as `chat_ingest` (so `ReviewService` is built **after** the pipeline in `main.py`); the
  narrow `ChatCaptureIngest` Protocol is declared in the services layer so it needn't import the
  chat package.
- **Conversation-time anchoring for agree.** ADR-048 ¬ß1 wants the capture's `created_at` = the
  anchoring message time (not the review-decision time). The distiller now computes the anchor for
  the **unclear** branch too (same `_anchor_time` the endorsed branch uses) and records it as
  **`anchor_at`** (ISO-8601) in the `stance-candidate` payload; agree reads it back
  (`datetime.fromisoformat`, fallback to the review item's `created_at`). A timestamp, never a node
  id ‚Äî so it can't strand the item across a reprocess (¬ß7's constraint is about ids). This is a
  build-detail refinement of ¬ß7's payload, recorded in [02-data-model](../02-data-model.md) +
  [03-api](../03-api.md); ADR-048 is left immutable (repo rule ‚Äî ADRs are superseded, never edited).
- **Kind-aware reprocess reset** (`PgReprocessStore.reset_derived_and_review`, refines ADR-042 ¬ß2):
  `TRUNCATE review_queue` ‚Üí `DELETE FROM review_queue WHERE kind <> 'stance-candidate'`. Stance
  items are **preserved** (their backing chat sessions are never touched by capture replay, and the
  watermark won't re-file them ‚Äî a parked human decision must survive); entity-ambiguity /
  vocab-proposal / dedup-proposal are cleared (re-derivable by replay + the nightly sweeps). Module
  + method docstrings updated to match.

### Independent review ‚Äî APPROVE-WITH-MINORS (no must-fix)

A fresh independent agent re-derived the requirements from ADR-048 ¬ß1/¬ß5/¬ß7 + the plan + 03-api +
CLAUDE.md, then checked the working-tree diff against the surrounding source. Confirmed all three
parts faithful ‚Äî the `ANY()` guard SQL, agree = the *same* `create_chat_capture` (one path, rule
2b/10), materialize-before-transition leaves a failed agree decidable (rule 7), the `anchor_at`
round-trip is the only way to stamp conversation-time without re-reading past-watermark messages
(faithful to ¬ß1), and the kind-aware DELETE preserves stance / clears the rest. **No must-fix.**
Minors: (1) `review.py` list docstring didn't mention the new kinds ‚Äî **fixed**; (2) the
`smoke_reprocess` reset mirror hardcoded the `'stance-candidate'` literal ‚Äî **fixed** (imports
`KIND_STANCE_CANDIDATE`; the inline mirror itself is unavoidable ‚Äî the store method opens its own
committing connection that a rolled-back txn can't contain); (3) the ADR-048 ¬ß7 payload list drifts
from 03-api on `anchor_at` ‚Äî **left as-is** (ADRs are immutable; the living contract is 02/03 + this
log); (4) unconfigured-ingest agree ‚Üí `BadResolution`/400 rather than 500 ‚Äî **left as-is**
(deliberately matches the existing vocab None-guard pattern; `# pragma: no cover`, always wired).

## Task 3

**`POST /chat/sessions/{id}/remember` + `POST /review/batch`** ([ADR-048](../adr/048-m6-chat-distiller-build-decisions.md)
¬ß6/¬ß8). Both endpoints are thin over existing services (rule 5); no migration (both reuse M6 task-1/2
tables). **Committed locally, not pushed.**

### What was built

- **`remember` ‚Äî on-demand distill of one session** (ADR-048 ¬ß6). `ChatDistillerService.remember(
  session_id)` runs the **same** `_distill_session` pass the nightly uses ‚Äî byte-identical salience
  gate, hedge downgrade, and stance routing, so a manual trigger changes *timing, not judgment*
  (**no force-endorse**). It advances the **same** `chat_distill_state` watermark, so it is idempotent
  with the nightly run and with a double-remember (the endorsed branch's deterministic capture id
  collapses a re-materialize ‚Äî rule 6). Opens a visible `agent_runs` row (vision P8, best-effort at
  row-open ‚Äî rule 7); endorsed captures organize **in the background** via the capture pipeline's
  fast-ack path, so the request only blocks on the one synchronous `conspect` call ¬ß6 mandates.
- **`PgChatDistillStore.session_state(session_id)`** ‚Äî the by-id, **no-idle-filter** lookup `remember`
  needs (the nightly's `distillable_sessions` filters by idle-eligibility, which `remember` bypasses).
  A LEFT-JOIN over `chat_sessions`/`chat_distill_state`/`chat_messages` grouped on the session +
  scalar watermark returns: **no row ‚Üí unknown session ‚Üí 404**; a row with `newest_at=NULL` ‚Üí known
  but message-less ‚Üí **skip**; else the watermark + newest for the delta. A new `SessionDistillState`
  carries the optional `newest_at`.
- **`remember` response mapping** (`app/routers/chat.py`) ‚Äî `200 {endorsed,to_review}` when the pass
  ran (incl. `0/0` for a zero-candidate/pure-retrieval chat ‚Äî it *ran*, not a skip), `200
  {skipped:reason}` when the watermark did not advance (no new messages / conspect chain down); `404`
  unknown session; `422` malformed id (uuid path). New `RememberResponse` model (either-or via
  nullable fields, matching 03-api's two shapes).
- **`POST /review/batch`** (ADR-048 ¬ß8) ‚Äî `ReviewService.resolve_batch(ids, action)` applies one
  `action` to many items **best-effort per item** (every id gets a `{id, ok, error?}`; one bad item
  never aborts the batch ‚Äî rule 7), reusing the **single-item `resolve` unchanged** (rule 10): the
  action is passed as **both `choice` and `verdict`**, and each kind's resolver reads only the field
  that fits it (`choice` for entity picks, `verdict` for stance/vocab) ‚Äî so a homogeneous batch (the
  common case: triaging stance candidates) resolves cleanly and a mismatched action fails just that
  item. Router route declared **before `/{review_id}`** so `/review/batch` isn't captured as a
  malformed-uuid path param; **config-capped** by a new `review_batch_max` (rule 9) ‚Üí `422` before any
  side effect. New `ReviewBatchRequest`/`ReviewBatchResponse`/`ReviewBatchResultItem` + a
  `BatchItemResult` service dataclass.
- Wiring: `ChatDistillerService` is now built in `main.py` **after** the capture pipeline (its
  endorsed branch is the pipeline's `create_chat_capture`, the single writer ‚Äî ADR-048 ¬ß1) and exposed
  via `app.state.chat_distiller_service` + a `get_chat_distiller_service` dependency. The standalone
  CLI/pipeline `build_chat_distiller_service` is untouched.

### Independent review ‚Äî APPROVE-WITH-MINORS (no must-fix)

A fresh independent agent re-derived the contract from ADR-048 ¬ß5‚Äì¬ß8 + 03-api + CLAUDE.md and checked
the diff. Confirmed: `remember` runs the same pass + same gate (no force-endorse), advances the same
watermark, and the three idempotency races (remember‚Üínightly, double-remember, crash-between-capture-
and-watermark) are all safe via the deterministic capture id + watermark; the `session_state` LEFT-JOIN
correctly separates 404 / skip / has-delta; the endorsed‚Üíorganize stays background; batch is declared
before `/{review_id}`, is best-effort, and the "action as both choice+verdict" is **safe across all
kinds** (each resolver reads one field, a mismatch fails only that item ‚Äî no wrong-success). **No
must-fix.** Two worthwhile minors **applied**: (1) `review_batch_max` config cap on the batch (rule 9,
matching `review_list_max`/`backfill_max_links`) ‚Üí `422`; (2) a smoke check for the `session_state`
zero-message (`newest_at=NULL`) branch. Logged, not actioned: the `RememberResponse` null-both-fields
serialization (contract-compatible, cosmetic); a mixed-kind batch's cross-kind action semantics (the
task-7 UI should scope batch to homogeneous selections). **Pre-existing observation (task-1 shared
path, out of scope):** `_distill_session` advances the watermark even when `_file_stance_candidate`
swallowed a review-store hiccup (rule-7 best-effort) ‚Äî the chat raw survives but that one unclear
candidate won't re-emit; worth a future confirm against the data-survival principle.

## Task 4

**One-tap remove for chat-distilled nodes** ([ADR-048](../adr/048-m6-chat-distiller-build-decisions.md)
¬ß11/¬ß12): the audit list + the first user-facing node removal ‚Äî the trust loop that lets the user
see what was auto-added overnight and undo any of it. **Committed locally, not pushed.**

### What was built

- **Migration 014** (plain SQL, ADR-011): **`captures.removed_at`** (a nullable tombstone ‚Äî replay-
  excluded) + the **`chat_auto_recorded`** registry (`capture_id` pk fk‚Üí`captures` `ON DELETE
  CASCADE`, `salience`, `recorded_at`). Additive + reversible (down/up round-trip verified on local
  pg16). The registry is the design's load-bearing distinction: **a row's existence marks a memory
  as auto-endorsed** ‚Äî so the audit list + remove are auto-endorsed only, and an *agree-from-review*
  memory (same `source=chat` capture, but no registry row) stays off the surface (ADR-048 ¬ß11). It
  is provenance, **not** derived index (a reprocess replays captures, not chat sessions, so it never
  re-mints these rows) ‚Äî recorded in [02-data-model](../02-data-model.md).
- **`app/chat/auto_recorded.py`** ‚Äî a new `AutoRecordedService` + `PgAutoRecordedStore`:
  - **audit list** (`GET /chat/auto-recorded`): the store's `chat_auto_recorded ‚ãà captures ‚ãà nodes`
    query, filtered `captures.removed_at IS NULL`, newest-first. A `LEFT JOIN LATERAL` picks the
    **primary content node's** title (entity-hub types excluded ‚Äî a minted hub isn't the memory);
    `snippet` is the whitespace-collapsed capture raw; `node_paths`/`source_ref`/`salience`/
    `created_at` ride the capture + registry. `title` is `NULL` until the background organize lands.
  - **remove** (`POST ‚Ä¶/remove`): compute the **content paths** (`node_paths` minus hub folders,
    ADR-038 ‚Äî a shared hub is never deleted, or every other node's edge into it dangles) ‚Üí git-rm
    them (`NodeWriter.remove_nodes`) ‚Üí **DB-delete the same content paths** (`chunks`/`edges` cascade
    both directions) ‚Üí **tombstone** the capture (`removed_at`) ‚Üí request a store commit. The DB
    delete is keyed to the content paths (not the unlink result) and the tombstone is stamped
    **last**, so a crash between the unlink and the tombstone self-heals on retry (re-unlink is a
    no-op on missing files, re-delete a no-op on absent rows) ‚Äî never a half-removed, unretryable
    state. `404` for a non-live-auto-recorded id (unknown / already-removed / not auto-endorsed).
- **Distiller wiring** ‚Äî the endorsed branch records the auto-endorse in the registry (`capture_id`
  + coarse `salience`) via a narrow `AutoRecordSink`. **Best-effort** (rule 7): a registry hiccup
  costs only the one-tap-remove affordance, never the endorse (the capture is the durable record).
  Idempotent (the deterministic capture id + `ON CONFLICT DO NOTHING`). The agree-from-review path
  in `ReviewService` is deliberately **not** wired to the sink (auto-endorsed only, ¬ß11).
- **Reprocess/replay exclusion** ‚Äî `PgReprocessStore.capture_ids_chronological` + `counts` skip
  `removed_at IS NOT NULL`, so `reprocess-all` never resurrects a removed memory (the headline M6
  Accept criterion). Plus a runtime guard in `CapturePipeline._replace_notes_via_reorganize`: the
  admin reorganize, a follow-up Pass-2, **and the ¬ß10 inbox drainer** (a later M6 task that drives
  `reorganize_capture`) all skip a tombstoned capture ‚Äî closing the review-found resurrection vector.
- Wiring: a shared `PgCaptureStore` (reused by the remove op's `node_paths` lookup); `AutoRecordedService`
  on `app.state` + a `get_auto_recorded_service` dependency; config `chat_auto_recorded_list_max` (rule 9).

### Independent review ‚Äî APPROVE-WITH-MINORS

A fresh independent agent re-derived the contract from ADR-048 ¬ß11/¬ß12 + the plan + 03-api + 02 +
CLAUDE.md and checked the diff. Confirmed: hub preservation (ADR-038), the reprocess-all exclusion,
the auto-endorsed-only scope gate (agree-from-review writes no registry row ‚Üí `is_recorded` false ‚Üí
404), the response shape + 404/422 mapping, and the additive/reversible migration. Findings, all
**resolved**:

1. **Must-fix ‚Äî resurrection vector.** `_replace_notes_via_reorganize` (admin reorganize / follow-up
   Pass-2, and soon the ¬ß10 inbox drainer) only bailed on `record is None`, so re-organizing a
   **tombstoned** capture's retained raw would rebuild the removed memory ‚Äî contradicting ¬ß11's
   replay-exclusion. **Fixed**: it now skips a capture with `removed_at` set (run ends `skipped`),
   the same exclusion `reprocess-all` applies; regression-tested (`test_reorganize_refuses_a_removed_capture`).
2. **Correctness minor ‚Äî remove not self-healing.** The DB delete was keyed to the paths this call
   actually unlinked, so a crash between the unlink and the DB delete stranded orphaned index rows on
   retry (files gone ‚Üí nothing re-deleted). **Fixed**: the delete targets the capture's content paths
   unconditionally (a no-op on absent rows); the idempotency test now asserts the DB delete still runs
   when the files are already gone.
3. **Minor ‚Äî type nit.** `_record_auto_endorsed(salience: str)` ‚Üí `str | None` (matches the sink). **Fixed.**

## Task 5

**Dedup sweep + extracted merge-core** ([ADR-049](../adr/049-dedup-sweep-merge-core-build-decisions.md)).
The nightly all-source near-duplicate sweep ‚Äî recently-ingested **content** nodes that read as
near-duplicates file a `dedup-proposal` the user resolves with **merge / keep / link** ‚Äî plus the
**merge-core extraction** the ADR pivots on. **Committed locally (`fd18c7b`), not pushed.**

### Merge-core (ADR-049 ¬ß1)

New `app/entities/merge_core.py` ‚Äî `MergeCore.fold(loser, survivor, reason, survivor_extra_paths=())`
= **retarget** the loser's inbound canonical edges ‚Üí survivor across their source files ‚Üí **tombstone**
the loser (`merged_into: survivor`) ‚Üí **reindex** the touched files ‚Üí **force commit+push**. The store
is truth (rule 1): it rewrites *files* and lets the indexer re-materialize edges/tombstone; per-file
misses are skip-and-continue (rule 7); no run/LLM (the caller owns `agent_runs`).
- **Entity-merge** (`MergeService`) refactored onto the core: it computes the alias union, `set_aliases`
  the survivor, then `fold(..., survivor_extra_paths=[survivor.path])`. **Behaviour-identical** ‚Äî the
  alias write simply moves ahead of the retarget (disjoint frontmatter regions); `test_entity_merge`
  asserts the same on-disk end state.
- **Content-merge** (dedup) is `fold(...)` **alone** ‚Äî no alias union; cross-content-type merges allowed.

### Dedup sweep (ADR-049 ¬ß3‚Äì¬ß7)

New `app/dedup/` ‚Äî `store.py` (`PgDedupStore`: candidate SQL + re-file guard + survivor degree) +
`sweep.py` (`DedupSweepService` + pure `default_survivor` + `build_dedup_sweep_service`). The candidate
SQL mirrors `compute_similar` (an HNSW **top-K LATERAL** per recent driver) then applies the **strict AND**
gate in SQL: **high cosine** (`nodes.embedding`) ‚ãÄ a **shared canonical edge to a common entity hub**
(an `array_agg` over `SELECT DISTINCT` shared hubs ‚Äî empty ‚áí NULL ‚áí excluded) ‚ãÄ **occurred-overlap** (a
null `occurred_start` on either side never excludes; the signal is True only when both dated and their
windows overlap). Content nodes only (`node_types` ‚àí `entity_like_types`). Only the **driver** is
recency-bounded (`indexed_at ‚â•` the last-successful-run watermark off `agent_runs`, fallback
`now ‚àí dedup_window_days`); pairs canonicalized `(least, greatest)` + de-duplicated, capped at
`dedup_max_pairs_per_run`. A **re-file guard** (`proposal_exists` on the canonical `payload->>` pair, any
status) makes a run-overlap re-scan idempotent. `default_survivor` = higher canonical degree, then older
`node_created_at` (fallback `indexed_at`), then least id. Payload = `{node_a, node_b, signals:{cosine,
shared_entity_ids, shared_entity_titles, occurred_overlap}, default_survivor}`. **No migration.** Config
knobs `dedup_min_cosine` (0.90) / `dedup_candidate_k` (10) / `dedup_window_days` (1) /
`dedup_max_pairs_per_run` (200) (rule 9); `dedup-sweep` CLI verb + `app.state` wiring (pipeline step =
task 8). The sweep only **files** items (no store writes) ‚Äî the actions happen at resolution.

### Resolution (ADR-049 ¬ß2/¬ß6)

`ReviewService` gained a `dedup-proposal` branch + `action`/`survivor` params (threaded through
`POST /review/{id}` + `ReviewResolveRequest`; the batch path passes `action` too): **merge** folds the
loser into the survivor (request `survivor` else `payload.default_survivor`) via the shared `MergeCore`

derived recompute (which only wipes `origin='derived'`), unlike a derived `similar`. `dedup-proposal` is
already truncated by the kind-aware reprocess reset (task 2), so a reprocess re-derives surviving dups.

## Task 6

**Inbox drainer** ([ADR-048](../adr/048-m6-chat-distiller-build-decisions.md) ¬ß10). The nightly job
that re-runs the **existing** `reorganize_capture` over captures still materialized as an `inbox/`
fallback (organize was down / produced no valid nodes at ingest), so a now-richer entity registry can
resolve a previously unorganizable capture into real typed nodes. Notes are **replaced only on
success**; a still-failing capture keeps its `inbox/` node and re-qualifies next run. **Committed
locally (`af14de5`), not pushed.**

### The job (`app/inbox/drain.py`)

`InboxDrainService.run_scheduled` mirrors the dedup-sweep template: opens its own `inbox-drain`
`agent_runs` row (vision P8), drains, closes it; **never raises** (rule 7). `_drain` selects the
inbox-materialized captures (bounded, oldest-first), and for each drives the pipeline **inline**, then
re-fetches to tally **resolved** (node_paths replaced out of `inbox/`) vs **still_inbox** (kept). Per
capture is **best-effort** ‚Äî a surprise error is caught (`errored++`) and never aborts the sweep. The
`InboxDrainOutcome` (found / reorganized / resolved / still_inbox / errored / truncated) feeds the run
summary + details. Depends on narrow protocols (`InboxCaptureReader`, `Reorganizer`, `AgentRunStore`)
so it unit-tests against fakes.

### The seams it reuses

- **`CapturePipeline.reorganize_capture_now`** ‚Äî the **blocking** reorganize the drainer drives (vs the
  admin path's background `_spawn`), so the job can bound the run + tally accurate outcomes and the
  CLI/pipeline drains deterministically. Same shared core (`_replace_notes_via_reorganize`): replace
  only on success, keep on the inbox fallback (marks the capture `failed` retryably), **skip a
  one-tap-removed capture** (`removed_at`), own `agent_runs` row. Rule 2b holds ‚Äî reorganize goes
  through the single writer.
- **`PgCaptureStore.list_inbox_materialized(folder, limit)`** ‚Äî `EXISTS (unnest(node_paths) LIKE
  folder/%)` inbox predicate **‚ãÄ `removed_at IS NULL`** (the ¬ß11 replay exclusion, so a removed memory
  can't resurrect through the drainer ‚Äî the same must-fix task 4 closed for the admin path) **‚ãÄ
  oldest-first `LIMIT`**. Deliberately **status-agnostic**: a prior drain may have re-marked a
  still-unresolvable capture `failed` while keeping its inbox node, and it must stay eligible for the
  next night's retry (not silently dropped ‚Äî rule 6).
- **`inbox_drain_max_per_run`** config (default 200, rule 9) bounds one run; the remainder waits for the
  next (the scan is idempotent). CLI verb **`inbox-drain`** (run-now + local-test), which flushes the
  store backup with `backup_now` so the one-shot commits resolved nodes (the in-app nightly's long-lived
  debounce handles that itself ‚Äî the reprocess-all pattern). Pipeline wiring is **task 8**.

Residual entity ambiguity files the normal `entity-ambiguity` items ‚Äî **no new review kind** (ADR ¬ß10).

## M6 follow-up (step-status fidelity) ‚Äî [ADR-050](../adr/050-pipeline-step-status-is-the-jobs-own-run.md)

**Why.** The live Accept `pipeline nightly` (parent `9d035ad5`) reported **`chat-distiller`** and
**`inbox-drain`** as **failed** ‚Äî 8 succeeded / 2 failed / 1 skipped. Root-caused (grilled
decision-by-decision) to a **status-fidelity defect, not data loss**: a capture that degrades to the
`inbox/` fallback closes its nested `agent="capture"` organize run **`failed`** (rule 7); because
`child_run_scope` flattens that nested run into the enclosing step, `_step_status` (any child not in
`{succeeded,skipped}` ‚Üí step failed) failed the whole step even though the distiller endorsed 4 and the
drainer correctly left 2 captures still-inbox (P10 held; the pipeline continued under `continue`). The
separate `claude` Max CLI 300s hang before Nebius fallback (nudge, 15:11:33) was a *post-pipeline* drain
event, **not** the cause of the failed steps.

**Fix (ADR-050, server-only, no migration).** `pipeline._step_status` now derives a step's status from
its **own** job run ‚Äî the child whose `agent == step.name` (an existing invariant: the step name *is* the
`agent_runs.agent` id). Nested spawned runs (`agent="capture"`) stay parented to the pipeline parent and
feed-visible (rule 7) but **do not gate** the step. `raised ‚Üí failed`, `halt`-on-own-failure, and
`store-sweep ‚Üí skipped` (opens no own run) are preserved; the step now reports its **own** `child_run_id`.
The inbox-fallback organize run keeps `status=failed` ‚Äî the fix lives only at the step-rollup layer.
