## Task 1

**`chat_distill_state` migration + the chat-distiller job** ([ADR-048](../adr/048-m6-chat-distiller-build-decisions.md)).
The stance-gated distiller that turns idle in-app chat sessions into memories — "the sleep cycle"
for chat. **Not yet scheduled** (it becomes a `nightly` pipeline step in task 8); a standalone
`python -m app.cli chat-distill` verb is the run-now + local-test path. **Committed locally, not
pushed.**

### What was built

- **Migration 013** (plain SQL, ADR-011): the **`chat_distill_state`** watermark table (`session_id`
  pk → `chat_sessions` `ON DELETE CASCADE`, `last_message_at`, `distilled_at`, nullable `run_id`) +
  a nullable **`captures.source_ref`** column. ADR-048 §1 mandates the endorsed capture carry
  `source_ref = <session-id>`, but the table had no such column and 02-data-model's captures
  paragraph didn't list one — a doc/ADR gap. Resolved by adding the column (mirrors `nodes.source_ref`)
  and aligning [02-data-model](../02-data-model.md) (not a replan: the ADR already *decided* it; only
  the mechanic was unstated). Additive + reversible (up/down/up round-trip verified on local pg).
- **`PgChatDistillStore`** ([app/chat/distill_store.py]) — the idle-eligibility query
  (`GROUP BY`/`HAVING max(created_at) < cutoff AND (watermark IS NULL OR max > watermark)` over a
  `LEFT JOIN`), **oldest-first** delta-after-watermark, and the watermark upsert.
- **`ChatDistillerService`** ([app/chat/distiller.py]) — one `conspect` pass over a session's
  fenced delta returns `{candidate_text, stance, salience, evidence_excerpt, referenced_entity_names,
  why_unclear}` candidates; the stance gate routes **endorsed → a `captures` row** (`source=chat`,
  `source_ref=session-id`, `created_at` = the anchoring message's time, via the capture pipeline =
  the single writer, rule 2b) → organizer; **unclear → a `stance-candidate` review item** (names +
  text, never node ids — ADR-048 §7); **rejected → run-log only**. Unknown stance biases to
  `unclear` (never guessed to endorsed); a light **hedge post-check** downgrades hedged "endorsed"
  to review; within-run **content dedup** guard; per-session best-effort; every run in `agent_runs`.
- **`CapturePipeline.create_chat_capture`** — the endorsement→organizer bridge, with a
  **deterministic capture id** (uuid5 over session + normalized statement) so a re-distill collapses
  instead of duplicating (rule 6). Extracted a shared **`build_capture_pipeline`** factory (rule 10)
  now used by both `reprocess.py` and the distiller CLI. Generalized the MCP burst semaphore to a
  `_organize_burst` shared by MCP + chat.
- **`KIND_STANCE_CANDIDATE`/`KIND_DEDUP_PROPOSAL`** constants; config knobs (`chat_distill_idle_hours`
  = 12h, `_max_sessions_per_run`, `_max_delta_messages`, `_max_candidates`); the `chat-distill` CLI verb.

### Independent review — APPROVE after fixes (2 must-fix, both resolved)

A fresh reviewer read ADR-048 + the docs + the full diff. Confirmed contract fidelity (one captures
row/endorsed candidate, exact §7 payload, rejected→log-only, stance never guessed, run never crashes,
migration additive/reversible) and cleared the `captures.source_ref` decision + the `build_capture_pipeline`
refactor. Two **must-fix** idempotency/visibility findings, both fixed + regression-tested:

1. **Cross-run duplicate materialization** — a partial per-session failure (one candidate's capture
   write fails after earlier ones succeeded) left the watermark un-advanced → the next run re-emitted
   the already-written candidates as duplicate nodes. **Fixed** by the deterministic capture id +
   already-materialized skip in `create_chat_capture` (rule 6: retries are safe).
2. **Silent delta truncation** — an oversized delta kept the *newest* N and advanced the watermark
   past the *older* skipped messages, permanently stranding them (and the config promised
   "dropped-and-logged"). **Fixed** by switching the delta to **oldest-first bounded batches**,
   advancing the watermark only to the last message actually processed (so the remainder is deferred
   to the next run, and the run records `truncated`) — a deferral, never a silent skip.

Minors logged as follow-ups (accepted, not blocking): node frontmatter carries `source=chat` but not
`source_ref` (the capture row does — audit chain is node→capture→session); equal-timestamp watermark
tie (rare at µs resolution; strict `>` kept for idempotency); `_anchor_time` prefix match can pick a
wrong message (impact = the memory's timestamp only; sane fallback).

### Verification

- **639 unit tests** green (+20: distiller parse/fence/hedge/anchor + stance-gate + truncation +
  idempotency + capture-source), **ruff clean**.
- **Migration 013** up/down/up round-trip verified on local pg16.
- **Real-PG smoke** 86/86 (+7): the un-fakeable eligibility `GROUP BY`/`LEFT JOIN`, oldest-first
  delta, watermark upsert-in-place + re-eligibility, and `captures.source_ref` round-trip.
- **CLI end-to-end on real dev data + real LLM**: `chat-distill` distilled 6 idle sessions → 1
  endorsed → a `captures` row (`source=chat`, `source_ref=<session-id>`, node **dated to the
  conversation day** not the run time — anchor-time works) → organizer → memory + event nodes →
  indexed; a second run = **0/0 sessions** (delta-after-watermark idempotency). *(This mutated the
  local dev graph store: one real chat-distilled node + 6 session watermarks — reversible via git;
  not run against prod.)*

## Task 2

**review_queue M6 kinds** ([ADR-048](../adr/048-m6-chat-distiller-build-decisions.md) §7): the
maybe-reopen fix, the `stance-candidate` agree=auto-endorse path, and the kind-aware reprocess
reset. Service/store only — the web Review surface for these is task 7. **Committed locally, not
pushed.**

### What was built

- **Maybe-reopen (the load-bearing guard fix).** `PgReviewQueue.resolve` guarded `WHERE
  status='pending'`, which wrongly made `maybe` terminal. Now the guard is **decidable = `pending`
  ∪ `maybe`** (`WHERE status = ANY($::text[])`, a new `DECIDABLE_STATUSES` constant); `resolved`/
  `discarded` stay terminal. `ReviewService.resolve` mirrors it (`record.status not in
  DECIDABLE_STATUSES` → `ReviewNotPending`/409) and so does `FakeReviewQueue`. This is generic — an
  entity-ambiguity `maybe` is now re-openable too, matching the 02-data-model wording.
- **`stance-candidate` resolution** (`ReviewService._resolve_stance_candidate`, verdict
  `agree|disagree|maybe`): **agree → the exact auto-endorse path** — it calls the *same*
  `CapturePipeline.create_chat_capture` the distiller's endorsed branch uses (one ingest path, not
  two — rule 10 / rule 2b), passing only the candidate's clean prose as the capture raw (organizer
  re-resolves everything, ADR-048 §2), idempotent via the deterministic capture id (rule 6);
  **disagree → discarded** (logged, never a node); **maybe → parked**, re-openable. The pipeline is
  injected as `chat_ingest` (so `ReviewService` is built **after** the pipeline in `main.py`); the
  narrow `ChatCaptureIngest` Protocol is declared in the services layer so it needn't import the
  chat package.
- **Conversation-time anchoring for agree.** ADR-048 §1 wants the capture's `created_at` = the
  anchoring message time (not the review-decision time). The distiller now computes the anchor for
  the **unclear** branch too (same `_anchor_time` the endorsed branch uses) and records it as
  **`anchor_at`** (ISO-8601) in the `stance-candidate` payload; agree reads it back
  (`datetime.fromisoformat`, fallback to the review item's `created_at`). A timestamp, never a node
  id — so it can't strand the item across a reprocess (§7's constraint is about ids). This is a
  build-detail refinement of §7's payload, recorded in [02-data-model](../02-data-model.md) +
  [03-api](../03-api.md); ADR-048 is left immutable (repo rule — ADRs are superseded, never edited).
- **Kind-aware reprocess reset** (`PgReprocessStore.reset_derived_and_review`, refines ADR-042 §2):
  `TRUNCATE review_queue` → `DELETE FROM review_queue WHERE kind <> 'stance-candidate'`. Stance
  items are **preserved** (their backing chat sessions are never touched by capture replay, and the
  watermark won't re-file them — a parked human decision must survive); entity-ambiguity /
  vocab-proposal / dedup-proposal are cleared (re-derivable by replay + the nightly sweeps). Module
  + method docstrings updated to match.

### Independent review — APPROVE-WITH-MINORS (no must-fix)

A fresh independent agent re-derived the requirements from ADR-048 §1/§5/§7 + the plan + 03-api +
CLAUDE.md, then checked the working-tree diff against the surrounding source. Confirmed all three
parts faithful — the `ANY()` guard SQL, agree = the *same* `create_chat_capture` (one path, rule
2b/10), materialize-before-transition leaves a failed agree decidable (rule 7), the `anchor_at`
round-trip is the only way to stamp conversation-time without re-reading past-watermark messages
(faithful to §1), and the kind-aware DELETE preserves stance / clears the rest. **No must-fix.**
Minors: (1) `review.py` list docstring didn't mention the new kinds — **fixed**; (2) the
`smoke_reprocess` reset mirror hardcoded the `'stance-candidate'` literal — **fixed** (imports
`KIND_STANCE_CANDIDATE`; the inline mirror itself is unavoidable — the store method opens its own
committing connection that a rolled-back txn can't contain); (3) the ADR-048 §7 payload list drifts
from 03-api on `anchor_at` — **left as-is** (ADRs are immutable; the living contract is 02/03 + this
log); (4) unconfigured-ingest agree → `BadResolution`/400 rather than 500 — **left as-is**
(deliberately matches the existing vocab None-guard pattern; `# pragma: no cover`, always wired).

### Verification

- **646 unit tests** green (+7: stance agree/disagree/maybe-park-and-reopen/resolved-terminal/
  bad-verdict/unconfigured-ingest/anchor-fallback + the distiller `anchor_at` assertion), **ruff
  clean**.
- **Real-PG smoke** green: `smoke_db` **91/91** (+5 — a new `check_review_queue_reopen` drives the
  real `PgReviewQueue` `ANY()` guard: pending→maybe→resolved→terminal, un-fakeable) + `smoke_reprocess`
  **12/12** (+2 — the kind-aware DELETE preserves a seeded stance-candidate, clears an entity item,
  in a rolled-back txn so dev data is untouched).

## Task 3

**`POST /chat/sessions/{id}/remember` + `POST /review/batch`** ([ADR-048](../adr/048-m6-chat-distiller-build-decisions.md)
§6/§8). Both endpoints are thin over existing services (rule 5); no migration (both reuse M6 task-1/2
tables). **Committed locally, not pushed.**

### What was built

- **`remember` — on-demand distill of one session** (ADR-048 §6). `ChatDistillerService.remember(
  session_id)` runs the **same** `_distill_session` pass the nightly uses — byte-identical salience
  gate, hedge downgrade, and stance routing, so a manual trigger changes *timing, not judgment*
  (**no force-endorse**). It advances the **same** `chat_distill_state` watermark, so it is idempotent
  with the nightly run and with a double-remember (the endorsed branch's deterministic capture id
  collapses a re-materialize — rule 6). Opens a visible `agent_runs` row (vision P8, best-effort at
  row-open — rule 7); endorsed captures organize **in the background** via the capture pipeline's
  fast-ack path, so the request only blocks on the one synchronous `conspect` call §6 mandates.
- **`PgChatDistillStore.session_state(session_id)`** — the by-id, **no-idle-filter** lookup `remember`
  needs (the nightly's `distillable_sessions` filters by idle-eligibility, which `remember` bypasses).
  A LEFT-JOIN over `chat_sessions`/`chat_distill_state`/`chat_messages` grouped on the session +
  scalar watermark returns: **no row → unknown session → 404**; a row with `newest_at=NULL` → known
  but message-less → **skip**; else the watermark + newest for the delta. A new `SessionDistillState`
  carries the optional `newest_at`.
- **`remember` response mapping** (`app/routers/chat.py`) — `200 {endorsed,to_review}` when the pass
  ran (incl. `0/0` for a zero-candidate/pure-retrieval chat — it *ran*, not a skip), `200
  {skipped:reason}` when the watermark did not advance (no new messages / conspect chain down); `404`
  unknown session; `422` malformed id (uuid path). New `RememberResponse` model (either-or via
  nullable fields, matching 03-api's two shapes).
- **`POST /review/batch`** (ADR-048 §8) — `ReviewService.resolve_batch(ids, action)` applies one
  `action` to many items **best-effort per item** (every id gets a `{id, ok, error?}`; one bad item
  never aborts the batch — rule 7), reusing the **single-item `resolve` unchanged** (rule 10): the
  action is passed as **both `choice` and `verdict`**, and each kind's resolver reads only the field
  that fits it (`choice` for entity picks, `verdict` for stance/vocab) — so a homogeneous batch (the
  common case: triaging stance candidates) resolves cleanly and a mismatched action fails just that
  item. Router route declared **before `/{review_id}`** so `/review/batch` isn't captured as a
  malformed-uuid path param; **config-capped** by a new `review_batch_max` (rule 9) → `422` before any
  side effect. New `ReviewBatchRequest`/`ReviewBatchResponse`/`ReviewBatchResultItem` + a
  `BatchItemResult` service dataclass.
- Wiring: `ChatDistillerService` is now built in `main.py` **after** the capture pipeline (its
  endorsed branch is the pipeline's `create_chat_capture`, the single writer — ADR-048 §1) and exposed
  via `app.state.chat_distiller_service` + a `get_chat_distiller_service` dependency. The standalone
  CLI/pipeline `build_chat_distiller_service` is untouched.

### Independent review — APPROVE-WITH-MINORS (no must-fix)

A fresh independent agent re-derived the contract from ADR-048 §5–§8 + 03-api + CLAUDE.md and checked
the diff. Confirmed: `remember` runs the same pass + same gate (no force-endorse), advances the same
watermark, and the three idempotency races (remember→nightly, double-remember, crash-between-capture-
and-watermark) are all safe via the deterministic capture id + watermark; the `session_state` LEFT-JOIN
correctly separates 404 / skip / has-delta; the endorsed→organize stays background; batch is declared
before `/{review_id}`, is best-effort, and the "action as both choice+verdict" is **safe across all
kinds** (each resolver reads one field, a mismatch fails only that item — no wrong-success). **No
must-fix.** Two worthwhile minors **applied**: (1) `review_batch_max` config cap on the batch (rule 9,
matching `review_list_max`/`backfill_max_links`) → `422`; (2) a smoke check for the `session_state`
zero-message (`newest_at=NULL`) branch. Logged, not actioned: the `RememberResponse` null-both-fields
serialization (contract-compatible, cosmetic); a mixed-kind batch's cross-kind action semantics (the
task-7 UI should scope batch to homogeneous selections). **Pre-existing observation (task-1 shared
path, out of scope):** `_distill_session` advances the watermark even when `_file_stance_candidate`
swallowed a review-store hiccup (rule-7 best-effort) — the chat raw survives but that one unclear
candidate won't re-emit; worth a future confirm against the data-survival principle.

### Verification

- **665 unit tests** green (+19: remember endorsed/unclear/unknown-404/no-new-messages/no-messages/
  chain-down/idempotent-second-call; remember router counts/skipped/404/422; batch agree-all/best-
  effort/bad-action; batch router delegate/route-precedence/empty-422/malformed-422/over-cap-422),
  **ruff clean**.
- **Real-PG smoke** `smoke_db` **94/94** (+3 in `check_chat_distill`: `session_state` returns
  watermark+newest for a known session, `None` for an unknown one, and known-with-`newest_at=NULL` for
  a message-less session — the un-fakeable LEFT-JOIN branches).
- Both routes confirmed present in the built app's OpenAPI (`/api/v1/chat/sessions/{session_id}/remember`,
  `/api/v1/review/batch`).

## Task 4

**One-tap remove for chat-distilled nodes** ([ADR-048](../adr/048-m6-chat-distiller-build-decisions.md)
§11/§12): the audit list + the first user-facing node removal — the trust loop that lets the user
see what was auto-added overnight and undo any of it. **Committed locally, not pushed.**

### What was built

- **Migration 014** (plain SQL, ADR-011): **`captures.removed_at`** (a nullable tombstone — replay-
  excluded) + the **`chat_auto_recorded`** registry (`capture_id` pk fk→`captures` `ON DELETE
  CASCADE`, `salience`, `recorded_at`). Additive + reversible (down/up round-trip verified on local
  pg16). The registry is the design's load-bearing distinction: **a row's existence marks a memory
  as auto-endorsed** — so the audit list + remove are auto-endorsed only, and an *agree-from-review*
  memory (same `source=chat` capture, but no registry row) stays off the surface (ADR-048 §11). It
  is provenance, **not** derived index (a reprocess replays captures, not chat sessions, so it never
  re-mints these rows) — recorded in [02-data-model](../02-data-model.md).
- **`app/chat/auto_recorded.py`** — a new `AutoRecordedService` + `PgAutoRecordedStore`:
  - **audit list** (`GET /chat/auto-recorded`): the store's `chat_auto_recorded ⋈ captures ⋈ nodes`
    query, filtered `captures.removed_at IS NULL`, newest-first. A `LEFT JOIN LATERAL` picks the
    **primary content node's** title (entity-hub types excluded — a minted hub isn't the memory);
    `snippet` is the whitespace-collapsed capture raw; `node_paths`/`source_ref`/`salience`/
    `created_at` ride the capture + registry. `title` is `NULL` until the background organize lands.
  - **remove** (`POST …/remove`): compute the **content paths** (`node_paths` minus hub folders,
    ADR-038 — a shared hub is never deleted, or every other node's edge into it dangles) → git-rm
    them (`NodeWriter.remove_nodes`) → **DB-delete the same content paths** (`chunks`/`edges` cascade
    both directions) → **tombstone** the capture (`removed_at`) → request a store commit. The DB
    delete is keyed to the content paths (not the unlink result) and the tombstone is stamped
    **last**, so a crash between the unlink and the tombstone self-heals on retry (re-unlink is a
    no-op on missing files, re-delete a no-op on absent rows) — never a half-removed, unretryable
    state. `404` for a non-live-auto-recorded id (unknown / already-removed / not auto-endorsed).
- **Distiller wiring** — the endorsed branch records the auto-endorse in the registry (`capture_id`
  + coarse `salience`) via a narrow `AutoRecordSink`. **Best-effort** (rule 7): a registry hiccup
  costs only the one-tap-remove affordance, never the endorse (the capture is the durable record).
  Idempotent (the deterministic capture id + `ON CONFLICT DO NOTHING`). The agree-from-review path
  in `ReviewService` is deliberately **not** wired to the sink (auto-endorsed only, §11).
- **Reprocess/replay exclusion** — `PgReprocessStore.capture_ids_chronological` + `counts` skip
  `removed_at IS NOT NULL`, so `reprocess-all` never resurrects a removed memory (the headline M6
  Accept criterion). Plus a runtime guard in `CapturePipeline._replace_notes_via_reorganize`: the
  admin reorganize, a follow-up Pass-2, **and the §10 inbox drainer** (a later M6 task that drives
  `reorganize_capture`) all skip a tombstoned capture — closing the review-found resurrection vector.
- Wiring: a shared `PgCaptureStore` (reused by the remove op's `node_paths` lookup); `AutoRecordedService`
  on `app.state` + a `get_auto_recorded_service` dependency; config `chat_auto_recorded_list_max` (rule 9).

### Independent review — APPROVE-WITH-MINORS

A fresh independent agent re-derived the contract from ADR-048 §11/§12 + the plan + 03-api + 02 +
CLAUDE.md and checked the diff. Confirmed: hub preservation (ADR-038), the reprocess-all exclusion,
the auto-endorsed-only scope gate (agree-from-review writes no registry row → `is_recorded` false →
404), the response shape + 404/422 mapping, and the additive/reversible migration. Findings, all
**resolved**:

1. **Must-fix — resurrection vector.** `_replace_notes_via_reorganize` (admin reorganize / follow-up
   Pass-2, and soon the §10 inbox drainer) only bailed on `record is None`, so re-organizing a
   **tombstoned** capture's retained raw would rebuild the removed memory — contradicting §11's
   replay-exclusion. **Fixed**: it now skips a capture with `removed_at` set (run ends `skipped`),
   the same exclusion `reprocess-all` applies; regression-tested (`test_reorganize_refuses_a_removed_capture`).
2. **Correctness minor — remove not self-healing.** The DB delete was keyed to the paths this call
   actually unlinked, so a crash between the unlink and the DB delete stranded orphaned index rows on
   retry (files gone → nothing re-deleted). **Fixed**: the delete targets the capture's content paths
   unconditionally (a no-op on absent rows); the idempotency test now asserts the DB delete still runs
   when the files are already gone.
3. **Minor — type nit.** `_record_auto_endorsed(salience: str)` → `str | None` (matches the sink). **Fixed.**
