## Tasks 2 + 3 + 5 (combined — the graph rewrite)

**Combined by the user's call at kickoff.** Tasks 2 (store writer + rename + bootstrap), 3
(organizer v3 + entity resolution), 5 (indexer/search retarget) are one atomic ~1300-reference
rename entangled through the capture pipeline, and migration 005 had already broken the indexer
against the live schema — so slicing them apart would leave the suite red at a task boundary. Done
as one rewrite with a single independent review. Server commits `2798412` (+ review follow-ups
`c54fed2`), local, **not pushed**. 240 unit tests pass, `ruff check` green.

**Task 2 — store writer + backup + bootstrap.** New `graph/node_writer.py` (`NodeWriter`): writes
`<type>/<slug>--<shortid>.md` (memory prefixes the date), the full frontmatter contract
(id/type/created/occurred?/source/source_ref?/plane?/planes/tags/aliases?/disambig?/
organizer_version/edges), atomic + collision-safe; each node gets a **fresh uuid** (`source_ref`
carries the capture id — siblings share it, ids are distinct so the DB pk + edges stay unique).
`services/vault_backup.py → store_backup.py` (`VaultBackupService → StoreBackupService`): ADR-014
machinery behaviour-identical, bootstrap skeleton = the 9 node-type folders + `inbox/`, `.gitignore`
drops `.obsidian` (Obsidian gone), remote wired from `GRAPH_STORE_REPO` via a new
`GitClient.set_remote` (zero-touch cutover, ADR-031 §6). Wholesale **vault → graph-store rename**
across config/`.env.example`, `git_repo`, `backup_jobs` (agent `vault-backup → store-backup`,
R2 `vault/ → store/`), `scheduler`, `system_health` (`/health` leg `vault → store`), `cli`,
`reindex`, `capture_store` (`note_paths → node_paths`), `models`, all routers, `dependencies`,
`main`. Deleted `capture/notes.py` + `graph/renderer.py`.

**Task 3 — organizer v3 + entity resolution.** `capture/organizer.py` rewritten: prompt emits
typed nodes vs the 9-type vocab (no fit → `memory` + a `node_type` **vocab-proposal**), `occurred`
kept only as valid partial-ISO (never fabricated), entity **mentions** (`{name,type,rel,disambig}`)
validated against `ENTITY_LIKE_TYPES`/`EDGE_RELS` (unknown → dropped + proposal), injection hygiene
(capture behind hard delimiters, declared data). New `entities/` package: `store.py` (`PgAliasStore`
— exact-on-normalized candidate lookup over the `nodes.aliases` GIN, tombstones excluded) and
`resolver.py` (`EntityResolver`): per-mention **exact short-circuit** (1 candidate → auto-link, no
LLM), **mint-new** (0 candidates → thin-hub `NodeDocument`), **multi-candidate → LLM disambiguation**
with structured candidates gated by `ENTITY_MATCH_MIN_CONF` (choice/conf; below floor or chain-down
→ **`entity-ambiguity` review item + edge left pending, never guessed**), **intra-capture dedup**,
**entropy guard**. New `services/review_queue.py` (`PgReviewQueue.enqueue`) — the kind-generic queue
**write path** (task-4's read/resolve UI still open). Pipeline rewired: organize → resolve mentions →
build `NodeDocument`s (content nodes get fresh ids + resolved entity edges with `since = occurred ??
created`; minted entities appended) → write → index; vocab proposals filed; all resolution best-effort
(a resolver failure leaves nodes unlinked, never fails the capture — rule 2).

**Task 5 — indexer/search retarget.** `indexing/frontmatter.py`: parses the node contract incl. the
`edges:` block list + partial-ISO `occurred` → date range + `merged_into`; id falls back to a
path-stable uuid5. `indexing/store.py` (`NodeUpsert`, id-keyed upsert, `update_node_path` for moves,
`replace_canonical_edges` materializing only edges whose target exists). `indexing/indexer.py`:
id-keyed, **whole-file** content hash, two-phase (upsert all, then materialize edges so same-batch
targets satisfy the FK). `graph/store.py`+`service.py`: derived `similar` edges **DB-only** into
`edges(origin='derived')` (renderer deleted, no file writes). `search/store.py`+`service.py`:
`SearchHit`/`NodeRow` typed, `types` filter, `GET /nodes/{id}` returns edges (canonical+derived,
both directions) + `profile` (null until task 6) + tombstone; router **302-redirects** a tombstone
to its survivor. `chunk_note → chunk_node`; `sb:related`/co-capture stripping deleted.

**Verification.** Beyond the 240 unit tests: a **Pg-store smoke test against a real migration-005
pgvector DB** inserted node/chunk/edge/alias/review rows and exercised every plain-SQL path
(`PgIndexStore`/`PgSearchStore`/`PgGraphStore`/`PgAliasStore`/`PgReviewQueue`) — it **caught a real
bug**: `replace_canonical_edges` compared a str `dst_id` against asyncpg's `UUID` objects, so
canonical edges silently never materialized; fixed (`str(r["id"])`). The full capture→…→search flow
was **not** run end-to-end (no live claude-max/Ollama) — that is the M3 Accept (task 10).

**Independent review** (fresh agent, diff vs 08 §M3 + ADR-030/031/032 + 02/03/04 + CLAUDE.md):

**Follow-ups (from the combined review):**
- **Alias accretion + fuzzy matching** (reviewer #1, the substantive one): resolution is
  exact-on-normalized only, and a successful link does **not** append the new surface form to the
  matched entity's `aliases`, nor does the organizer elicit per-mention `aliases`. So "Alex" then
  "Alexandru" across captures → **two** `person` nodes. Repeated *identical* mentions collapse
  correctly. The literal M3 Accept ("mentioned twice → one node") holds for identical forms; the
  Alex/Alexandru case needs alias accretion (rewrite the entity's frontmatter aliases on link) +
  the organizer emitting aliases, and/or `pg_trgm` fuzzy matching (ADR-032 defers fuzzy). **Wire at
  task 6 (entity services) or before the task-10 Accept.**
- Surface entity-resolution outcomes (resolver `model_used`/`fallback_used`, pending count) into the
  capture's `agent_runs` details for observability (reviewer #3, rule 3/7). Review items are already
  persisted, so nothing is lost.
- Incremental `reindex_all` re-materializes canonical edges only for re-indexed (non-hash-skipped)
  nodes; near-unreachable today (thin entity targets can't fail to index), a DB-wipe reindex restores
  all — but tighten `replace_canonical_edges`'s "reconciled on next reindex" promise if the thin-target
  guarantee ever weakens (reviewer #2).
- Dead `EntityResolver` `since` param + unused `entity_alias_min_fuzzy_len`/`Mention.aliases` — remove/
  wire when fuzzy + accretion land (reviewer #5).

## Task 4

**Review-queue read/resolve surface** — server commit `c53eadc`, local, **not pushed**. 265 unit
tests pass (25 new), `ruff check` green, real-DB smoke on the new SQL passed.

The write path landed in task 3 (`PgReviewQueue.enqueue`); this adds the admin **read/resolve**
half over the same kind-generic `review_queue` table (03-api §Review, ADR-030 §3 / ADR-029).

**Store (`services/review_queue.py`).** Added `ReviewRecord` + a `ReviewReadStore` protocol
(`list_items`/`get`/`resolve`) alongside the existing write-only `ReviewQueue`; `PgReviewQueue`
implements both (plain SQL). `resolve` is a **guarded** `UPDATE … WHERE id=$1 AND status='pending'
RETURNING id` so a double-resolve is a no-op. Status constants added (`pending/resolved/discarded/
maybe`). jsonb payload/resolution decode tolerantly (text-or-dict, like `agent_runs`).

**Service (`services/review_service.py`, new).** `ReviewService` holds the resolution business
logic the router delegates to (rule 5). `list_items` (status defaults `pending`; `all`/empty drops
the filter). `resolve` dispatches by kind:
- **entity-ambiguity** — `choice = <candidate id>` materializes the pending edge; `"new"` mints a
  thin entity hub (title + alias, source `review`) then materializes onto it; `"maybe"` defers
  (status `maybe`, draws nothing). Materialization reuses the store's own machinery: `NodeWriter.
  add_edges` appends the edge to each source node's file (atomic, idempotent), then the **indexer
  re-reads that file** and materializes the canonical edge into `edges` (never a direct DB write —
  rule 1); a store commit is requested (ADR-014). For `"new"`, the minted entity + source files are
  indexed in **one** `index_paths` call so the two-phase indexer upserts the entity before the
  source edge's `dst_id` FK is checked. A source node that isn't indexed / has vanished is skipped,
  never fatal (rule 7).
- **vocab-proposal** — `verdict = "reject"` discards; `"approve"` records the verdict + opens a
  **queued `vocab-consolidation` marker run** (SKIPPED, `details.queued=true`, carrying `vocab`/
  `value`/`review_id`) for visibility (vision P8). **Task 4 only queues** — the retro-consolidation
  job that mutates the live vocabulary + re-walks the graph is **task 7**, which consumes these
  markers / the resolved vocab-proposal items (**user call at kickoff: verdict + queue-hook now,
  mutation in 7**).

**Write-path enrichment (the piece task 4 depended on).** An `entity-ambiguity` item filed by the
resolver did **not** carry the source node id the pending edge originates from (the resolver files
the item before content-node ids exist, and pending mentions are skipped, so nothing on disk records
them). Fix: the pipeline now **pre-assigns each content node's id** and builds a
`pending_edges_by_key` map (`{mention_key: [{src, rel, since}]}`) passed into `resolver.resolve`;
`_file_review` stores it as `payload.pending_edges`, and `_build_documents` reuses the same ids so
file ↔ review agree. Purely additive to the resolver/pipeline (optional kwarg; existing tests
unchanged). `since` matches what `_build_documents` stamps (`occurred ?? created`).

**Router (`routers/review.py`, new) + wiring.** `GET /review?status=&kind=` → list;
`POST /review/{id}` (id typed `uuid.UUID` → `422` on garbage, matching the activity router) →
`404` unknown / `409` already-resolved / `400` bad body for the kind. `NodeWriter` + `PgIndexStore`
are now hoisted in `main.py` and **shared** between the capture pipeline and the review service. New
setting `review_list_max` (200, rule 9). Contract-doc fields (`GET /review` returns `id/kind/payload
/excerpt/source/source_ref/created_at`) are all present; the response adds `status`/`resolution` (a
harmless superset the admin list uses).

**Verification.** 25 new unit tests (service resolve/list with fakes + a real `NodeWriter`; the
router status-code mapping; `append_edges`/`add_edges`; the resolver `pending_edges` enrichment). A
**real migration-005 pgvector smoke** exercised the new `PgReviewQueue` SQL end-to-end (enqueue →
list/kind-filter → get with `pending_edges` round-trip → guarded resolve → post-resolve list),
confirming the uuid-column str round-trip, jsonb encode/decode, and the pending-only guard. The full
capture→review→resolve→materialize→index flow end-to-end (live LLM) is part of the M3 Accept (task
10). **Independent structured review** (diff vs 08 §M3 + ADR-030/027 + 03-api + CLAUDE.md): no
must-fix findings — contract, layering, never-lose/visible, and store-is-truth all hold.

**Follow-ups:**
- **Task 7 (vocab surface)** consumes the queued `vocab-consolidation` markers (or the resolved
  `vocab-proposal` items) to mutate the live node/edge-type vocabulary + run the retro-consolidation
  walk. `GET /types` / `PUT /settings/vocabulary` land there too.
- `maybe` is terminal via the guarded transition (can't be re-resolved through this endpoint);
  M6's polished review UX revisits deferral/reopen if needed.
