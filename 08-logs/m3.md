## Task 4

**Review-queue read/resolve surface** — server commit `c53eadc`, local, **not pushed**. 265 unit
tests pass (25 new), `ruff check` green, real-DB smoke on the new SQL passed.

The write path landed in task 3 (`PgReviewQueue.enqueue`); this adds the admin **read/resolve**
half over the same kind-generic `review_queue` table (03-api §Review, ADR-030 §3 / ADR-029).

**Store (`services/review_queue.py`).** Added `ReviewRecord` + a `ReviewReadStore` protocol
(`list_items`/`get`/`resolve`) alongside the existing write-only `ReviewQueue`; `PgReviewQueue`
implements both (plain SQL). `resolve` is a **guarded** `UPDATE … WHERE id=$1 AND status='pending'
RETURNING id` so a double-resolve is a no-op. Status constants added (`pending/resolved/discarded/
maybe`). jsonb payload/resolution decode tolerantly (text-or-dict, like `agent_runs`).

**Service (`services/review_service.py`, new).** `ReviewService` holds the resolution business
logic the router delegates to (rule 5). `list_items` (status defaults `pending`; `all`/empty drops
the filter). `resolve` dispatches by kind:
- **entity-ambiguity** — `choice = <candidate id>` materializes the pending edge; `"new"` mints a
  thin entity hub (title + alias, source `review`) then materializes onto it; `"maybe"` defers
  (status `maybe`, draws nothing). Materialization reuses the store's own machinery: `NodeWriter.
  add_edges` appends the edge to each source node's file (atomic, idempotent), then the **indexer
  re-reads that file** and materializes the canonical edge into `edges` (never a direct DB write —
  rule 1); a store commit is requested (ADR-014). For `"new"`, the minted entity + source files are
  indexed in **one** `index_paths` call so the two-phase indexer upserts the entity before the
  source edge's `dst_id` FK is checked. A source node that isn't indexed / has vanished is skipped,
  never fatal (rule 7).
- **vocab-proposal** — `verdict = "reject"` discards; `"approve"` records the verdict + opens a
  **queued `vocab-consolidation` marker run** (SKIPPED, `details.queued=true`, carrying `vocab`/
  `value`/`review_id`) for visibility (vision P8). **Task 4 only queues** — the retro-consolidation
  job that mutates the live vocabulary + re-walks the graph is **task 7**, which consumes these
  markers / the resolved vocab-proposal items (**user call at kickoff: verdict + queue-hook now,
  mutation in 7**).

**Write-path enrichment (the piece task 4 depended on).** An `entity-ambiguity` item filed by the
resolver did **not** carry the source node id the pending edge originates from (the resolver files
the item before content-node ids exist, and pending mentions are skipped, so nothing on disk records
them). Fix: the pipeline now **pre-assigns each content node's id** and builds a
`pending_edges_by_key` map (`{mention_key: [{src, rel, since}]}`) passed into `resolver.resolve`;
`_file_review` stores it as `payload.pending_edges`, and `_build_documents` reuses the same ids so
file ↔ review agree. Purely additive to the resolver/pipeline (optional kwarg; existing tests
unchanged). `since` matches what `_build_documents` stamps (`occurred ?? created`).

**Router (`routers/review.py`, new) + wiring.** `GET /review?status=&kind=` → list;
`POST /review/{id}` (id typed `uuid.UUID` → `422` on garbage, matching the activity router) →
`404` unknown / `409` already-resolved / `400` bad body for the kind. `NodeWriter` + `PgIndexStore`
are now hoisted in `main.py` and **shared** between the capture pipeline and the review service. New
setting `review_list_max` (200, rule 9). Contract-doc fields (`GET /review` returns `id/kind/payload
/excerpt/source/source_ref/created_at`) are all present; the response adds `status`/`resolution` (a
harmless superset the admin list uses).

**Verification.** 25 new unit tests (service resolve/list with fakes + a real `NodeWriter`; the
router status-code mapping; `append_edges`/`add_edges`; the resolver `pending_edges` enrichment). A
**real migration-005 pgvector smoke** exercised the new `PgReviewQueue` SQL end-to-end (enqueue →
list/kind-filter → get with `pending_edges` round-trip → guarded resolve → post-resolve list),
confirming the uuid-column str round-trip, jsonb encode/decode, and the pending-only guard. The full
capture→review→resolve→materialize→index flow end-to-end (live LLM) is part of the M3 Accept (task
10). **Independent structured review** (diff vs 08 §M3 + ADR-030/027 + 03-api + CLAUDE.md): no
must-fix findings — contract, layering, never-lose/visible, and store-is-truth all hold.

**Follow-ups:**
- **Task 7 (vocab surface)** consumes the queued `vocab-consolidation` markers (or the resolved
  `vocab-proposal` items) to mutate the live node/edge-type vocabulary + run the retro-consolidation
  walk. `GET /types` / `PUT /settings/vocabulary` land there too.
- `maybe` is terminal via the guarded transition (can't be re-resolved through this endpoint);
  M6's polished review UX revisits deferral/reopen if needed.

## Task 6

**Entity services — merge propose/apply + tombstones, backfill scan, tiered profile-refresh** —
server commits `f5795fe` (merge + migration 006), `9e544c3` (backfill + profiles), `f6f0678`
(review follow-ups), local, **not pushed**. 294 unit tests pass (18 new), `ruff check` green.

The three ADR-030 §4/§5/§6 entity-lifecycle services. A shared `entities/entity_store.py`
(`PgEntityStore`) centralizes the DB reads all three need (node lookup, inbound canonical edges,
entity listing, `entities_touched_since`, 1-hop neighborhood, alias-substring match) — every read
excludes tombstones (`merged_into` set). All three keep the store as truth (rule 1): they rewrite
**files** and let the indexer materialize the DB, never writing the graph tables directly.

**Merge (`POST /admin/entities/merge`, ADR-030 §5).** Two-step, ADR-024 shape. `MergeService.propose`
(sync, no writes/LLM) validates the pair (self-merge → 400, unknown → 404, tombstone endpoint /
non-entity → 400) and returns the inbound-edge inventory. `apply` opens an `agent="entity-merge"`
run and, in the background (202 `{run_id}`): retargets every inbound canonical edge loser→survivor on
the **source files** (new pure `retarget_edges` in `graph/node_writer.py` — redirects `to:`, drops the
now-duplicate edge so the pk survives reindex), **unions** the survivor's aliases with the loser's
name + aliases (`merged_alias_union` + `upsert_frontmatter_list`), writes the loser's permanent
**tombstone** (`render_tombstone` — keeps only `id`/`type`/`merged_into`, same path so old locators
resolve), reindexes the touched files, and **force-commits** (`backup_now`) — immediate, git history
is the checkpoint (no agent-window wait). The tombstone 302-redirect + search/neighbor exclusion were
already in place (task 5). Per-file failures are skip-and-continue (rule 7).

**Backfill scan (nightly, ADR-030 §6).** `entities/backfill.py` (`agent="entity-backfill"`, 04:20):
entities touched since the last successful run (watermark = its `started_at`; first run = the
`BACKFILL_WINDOW_DAYS` window) are re-checked against recent memory nodes whose chunk text contains a
**qualifying alias** (length ≥ `ENTITY_ALIAS_MIN_FUZZY_LEN` — ADR-032 entropy guard; shorter aliases
never scanned) and that lack an edge to the entity. A match auto-adds an `involves` edge on the memory
file (feed-flagged in the run), then reindex + force-commit. `NOT EXISTS`-edge exclusion + the
watermark make re-runs duplicate-free; `add_edges` now returns whether it actually wrote so the link
count/budget aren't double-spent when one memory matches two aliases of the same entity.
**M3 scoping (recorded):** the ADR-030 §6 "below-threshold → review item" branch is deliberately
**narrowed** — a coarse substring backfill would file noisy review items on short aliases, so those
are skipped, not reviewed. The richer review-generating drain (inbox re-organization + dedup
proposals) is M6 (04 §4 (d)/(c)); this M3 job ships the high-confidence auto-link half.

**Profile-refresh (nightly, ADR-030 §4 / ADR-034).** `entities/profile_refresh.py`
(`agent="profile-refresh"`, 04:10) + pure `entities/profiles.py`. For every entity-like node: compute
the 1-hop neighborhood → **tier by graph degree** (`stub` < `PROFILE_TIER_SNAPSHOT_MIN`(3) ≤ `snapshot`
< `PROFILE_TIER_FULL_MIN`(8) ≤ `full`, ADR-034) → regenerate only when the neighborhood **hash**
changed (idempotency + structurally caps nightly LLM spend). Stub = mechanical categorized observation
lines carrying `(as of …)` stamps, **no LLM**; snapshot/full = an LLM synthesis (observations injected
as delimited DATA — titles/rels/dates only, never bodies; injection hygiene). The profile text is
embedded and stored. Best-effort (rule 7): LLM down → store the stub text + `stub` tier + cleared hash
so the next run retries; embed down → store without a vector. `GET /nodes/{id}` now serves `profile`
from `node_profiles` via a LEFT JOIN in the search store's `get_node` (one query, was stubbed `null`).

**Migration 006 + storage decision (implementer's — docs delegate it).** 02-data-model §3 leaves the
derived-profile storage "an implementation choice within the derived tier". Chosen: a dedicated
**`node_profiles`** table (`node_id` PK/FK cascade, `tier`, `profile` text, `observations` jsonb with
supporting node ids, `neighborhood_hash`, `embedding vector(768)`, `refreshed_at`) — a separate table
(not a `nodes` column) keeps the indexer the sole writer of `nodes`/`chunks` and the profile-refresh
job the sole writer of profiles (clean ownership); fully rebuildable (a wiped table repopulates on the
next refresh; a reindex doesn't touch it). `compute_head` advances to `006`.

**Wiring.** All three services on `app.state` + `main.py`; merge behind `get_merge_service` + the admin
router endpoint; the two nightly jobs registered in the scheduler (optional params) and the CLI
(`python -m app.cli profile-refresh` / `entity-backfill`). Shared `StoreCommitter` protocol lifted to
`store_backup.py`.

**Verification.** 18 new unit tests over pure logic (retarget/tombstone/alias-union; tier selection,
observation rendering, neighborhood-hash stability) + the services against fakes + a real `NodeWriter`
(merge rewrites, backfill auto-link + no-double-count + watermark, profile tiering/skip/degrade) + the
merge router status-code mapping. **Not yet run: a real migration-006 pgvector smoke** over the new SQL
(the `entity_store` queries, `profile_store` upsert, the `node_profiles` LEFT JOIN, migration 006
up→down→up) — unlike tasks 1/4/5 this was left to the M3 Accept (task 10) alongside the deferred
live-LLM end-to-end flow. **Independent review** (fresh agent, diff vs 08 §M3 + ADR-030/034 + 02/03/04
+ CLAUDE.md): **no must-fix findings** — merge/backfill/profile correctness, the contract, and the
invariants (store-is-truth, never-lose, never-crash, idempotency, config-via-settings) all hold. 5
minors: 3 fixed inline (`f6f0678` — backfill double-count; degraded-profile tier/content agreement;
stored-observation bound), 2 logged ↓.

**Follow-ups:**
- **Profile embedding not yet surfaced in search** (reviewer + design): `node_profiles.embedding` is
  populated but nothing reads it — an entity's current profile doesn't yet appear as a search hit
  ("searching Alex hits the summary", ADR-030 §4). Wire the profile embedding into search/`similar`
  (a small SQL union) at task 8/10 or with the M4 retrieval work.
- **Real migration-006 pgvector smoke** over the new SQL — run at/before the task-10 Accept (as the
  live-LLM flow is), verifying migration 006 up→down→up and every new plain-SQL path against a real DB.
- **Merge/backfill source-file rewrites aren't serialized under the store lock** (reviewer #4): each
  file write is atomic, but two concurrent applies (or an apply racing a capture) touching the *same*
  source file can lose a retarget (last-writer-wins). Matches the existing tag-consolidation-apply
  precedent (the store lock guards git only) and git history is the revert net, so it's a latent gap on
  infrequent manual admin ops, not a functional bug — revisit if apply/capture concurrency tightens.
- **Alias accretion** (the combined-review follow-up carried from tasks 2/3/5): resolution still does
  not append a newly-met surface form to a matched entity's `aliases`, so Alex/Alexandru across
  *organize* passes still mints two nodes until merged. Backfill now links *memories* to entities by
  alias, and merge collapses the two hubs, but the accretion-on-link half (+ the organizer emitting
  aliases) remains open — wire before the task-10 Accept.

## Task 7a

**Vocabulary governance core — `GET /types`, `PUT /settings/vocabulary`, forward-live effective
vocab** — server commit `dd3c5be`, local, **not pushed**. 317 unit tests pass (+23), `ruff check`
green. Scope split recorded at kickoff ([08 §M3](../08-implementation-plan.md) task 7 /
[ADR-035](../adr/035-vocabulary-consolidation-scope-m3.md)): **7a = the forward-live half**
(governance + approval makes a type live + a feed-visible consolidation run); **7b = the edge
retro-consolidation apply** (deferred). The retro-consolidation *mechanics* were under-specified in
ADR-027 §3 — grilled at kickoff, recorded as ADR-035 **before** coding (09 discipline), then built.

**Approved-vocabulary store (`vocab/store.py`).** New `PgVocabularyStore` over the existing
`app_settings` table (its first user), one `"vocabulary"` key → jsonb `{node_types, edge_rels,
entity_like_types}` = the user-approved **additions** per axis. `get_additions()` + `add()`
(read-modify-write in a `FOR UPDATE` txn → upsert; dedup, order-preserving, idempotent — re-approving
a type is a no-op). Pure decode/merge helpers unit-tested; the SQL itself needs a real-DB smoke (↓).

**Effective vocabulary (`vocab/service.py`).** `EffectiveVocabulary` = config **seeds ∪ approved
additions** per axis (02 §3 "config + app_settings"). `effective_vocabulary(vocab, settings)` is the
one accessor every writer reads through; a `None` provider falls back to the seeds, so the ~6
services that read vocabulary gained an **optional** `vocab` param and existing tests construct
unchanged. Threaded into: the organizer prompt + `validate_organizer_output` (capture pipeline), the
entity **resolver** (entity-like filter), the nightly **backfill**/**merge**/**profile-refresh**
(entity-like listing), and the review-queue **mint** (a `new`-entity of a freshly-approved type is
now accepted). Result: an approved type is **forward-live at once** everywhere, not just in `/types`.

**The choke point (`VocabularyService`).** `list_types()` serves `GET /types` (effective vocab +
pending `vocab-proposal` items). `resolve_proposal(review_id, verdict)` is the **one** approve/reject
path behind both `PUT /settings/vocabulary` and `POST /review/{id}` (ADR-027 §4): approve writes the
type to `app_settings` (`node_type`→node_types; `entity_type`→**both** node_types + entity_like_types,
so `ENTITY_LIKE_TYPES ⊆ NODE_TYPES` holds for the effective sets; `edge_rel`→edge_rels), opens the
`vocab-consolidation` run, then flips the item `resolved`; reject discards. Mutation happens **before**
the run + status transition, so a failure leaves the item pending (retryable) — and every step is
idempotent. The **Review service delegates** its whole `vocab-proposal` branch here (it owns its own
transition); the three resolution exceptions moved to `review_queue` (shared, no service
cross-import), re-exported from `review_service` for the router.

**Consolidation run (`vocab/consolidation.py`, 7a slice).** `VocabConsolidation.start()` opens a
`SUCCEEDED` `vocab-consolidation` `agent_runs` row recording the now-live mutation (vision P8) —
replacing task 4's `SKIPPED` queued marker. Best-effort (rule 7): the vocab is already committed, so
a run-store hiccup never fails the approval. **7b** hangs the edge re-walk propose→apply off this same
