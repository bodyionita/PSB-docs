## Task 3

**Indexer service (real index step, replacing the M1 stub)** — server commit `684604e`, local,
**not pushed**. The `notes`/`chunks` index is now populated from the vault (04 §3,
[ADR-022](../adr/022-embeddings-self-hosted-nomic.md)/[ADR-023](../adr/023-semantic-relatedness-graph.md)):

- **`app/indexing/indexer.py`** — the `Indexer`. Per note: read → strip `sb:related` → **sha256
  `content_hash`** (skip if unchanged) → `parse_note_metadata` → `chunk_note` → **batch-embed a
  note's chunks in one `registry.embed` call** with the mandatory `search_document: {title}\n\n{chunk}`
  prefix (bounded backoff on `ProviderUnavailable`) → **per-note transaction** (upsert note + replace
  chunks) + `notes.embedding` = **mean-pool** of the chunk vectors. On embed failure:
  **skip-and-continue → `partial`**, existing rows left intact (a later reindex retries). Two entry
  points: `index_paths(paths)` (the capture/ingestion write path) and `reindex_all()` (full rescan +
  **deletion reconciliation**, cascade). The **relatedness graph is left untouched** — nightly-only
  per ADR-023. `content_hash` `strip()`s surrounding whitespace so the trailing `sb:related` block
  (the graph's own writes) never re-triggers a reindex.
- **`app/indexing/frontmatter.py`** — pure frontmatter parser + `NoteMetadata` extraction
  (plane←folder / planes←`[plane]` / tags / source / source_ref / created←mtime fallbacks; title =
  H1 else filename stem). **No YAML dependency** — the parser matches the controlled render shape and
  reads user notes leniently.
- **`app/indexing/store.py`** — `IndexStore` protocol + `PgIndexStore` (the per-note transaction in
  plain SQL, upsert on `vault_path`, `list_indexed_paths`/`delete_notes` for reconciliation).
- **`db.py`** — registers a **text codec for the pgvector `vector` type** on every pooled connection,
  so embeddings pass as plain `list[float]` (no `::vector` casts / string juggling in query code);
  encode on INSERT, decode on SELECT.
- **Capture pipeline wired** — the M1 no-op stub (both `_process` and the reorganize path) now calls
  `index_paths`. **Best-effort**: the vault is truth (rule 1) and the notes are already on disk, so
  an index/embed failure **never flips a written capture to `failed`** — it's logged and the outcome
  recorded to the capture's `agent_runs` `details.index` (ADR-021). New settings `EMBED_MAX_ATTEMPTS`
  / `EMBED_RETRY_BACKOFF_SECONDS` (rule 9).
- **Independent review (`/code-review` high, protocol-sanctioned mechanism): 1 must-fix, fixed.**
  `index_paths` promised "never raised — the rest of the batch still indexes," but `_index_one` only
  caught `ProviderUnavailable`; an unexpected per-note store/DB error would abort the whole batch
  (breaks rule 7 / "never crashes the job"). **Fixed:** the per-note loop now catches any exception,
  logs it, counts the note `failed`, and continues. Added coverage (batch survives an unexpected
  store error; pipeline indexes exactly the written notes + logs the outcome).
- **173 pytest + ruff green** (18→20 new across `test_frontmatter` + `test_indexer` +
  `test_capture_pipeline`). **Verified end-to-end against a real pgvector DB** (dev compose, at head
  004): `vector` codec round-trips (768-dim `list` in/out), per-note transaction lands note+chunks,
  mean-pool stored, hash-skip on unchanged reindex, cosine query runs over the HNSW index, and
  `reindex_all` reconciles a deleted file's row away.

## Task 5

**Relatedness graph recompute + `sb:related` render** — server commit `73ed641`, local, **not
pushed**. The topical-relatedness graph ([ADR-023](../adr/023-semantic-relatedness-graph.md), 02 §3,
04 §3) — a new `app/graph/` domain package with one entry point, `RelatednessGraph.recompute()`:

- **`app/graph/store.py`** — `GraphStore` protocol + `PgGraphStore` (plain SQL, ADR-011).
  `compute_neighbors` = a per-note **LATERAL top-K nearest** over `notes.embedding` cosine, the
  `RELATED_MIN_SCORE` floor applied **after** the top-K cut (so a note keeps *at most* K, possibly
  fewer), `m.id <> n.id` (no self-links), NULL embeddings excluded. `replace_note_links` rewrites the
  whole **canonical** `note_links` table in one transaction. Distance written `m.embedding <=>
  n.embedding` (indexed operand left, HNSW-friendly) + an `m.id` **deterministic tiebreaker** on both
  the top-K cut and the render order, so exact-score ties can't reorder a block and spuriously churn
  the vault.
- **`app/graph/renderer.py`** — pure, **idempotent** render/apply of the machine-owned `sb:related`
  `## Related notes` block: path-target + title-alias `[[wikilinks]]` (title falls back to the
  basename), placed at the end of the body, LF-normalized, strips a **stale** block when a note loses
  all neighbours. Strictly distinct from the co-capture `related:` frontmatter / human `## Related`
  section (untouched).
- **`app/graph/service.py`** — `recompute()`: **always** replaces `note_links` (canonical, reflects
  current embeddings even at zero vault churn) → renders every indexed note's block **churn-gated**
  (rewrite only when the file content actually changed → a stable graph = **zero** writes, ADR-023 §5)
  → **atomic** temp+`os.replace` writes → requests a vault commit via the `VaultBackup` seam **only
  when ≥1 file changed** (the single git owner folds it into one commit+push under the vault lock; the
  nightly job/`/admin/reindex` wrapper is Task 6). The feedback-loop fix holds: the indexer's
  `content_hash` excludes the block, so these writes never re-trigger a reindex. `RELATED_TOP_K`=5 /
  `RELATED_MIN_SCORE`=0.5 settings (rule 9, tuned live at Accept). Wired onto `app.state` + a
  `get_relatedness_graph` accessor for Task 6.
- **Independent review (protocol-sanctioned, fresh context): 1 must-fix, fixed.** A
  present-but-unreadable/unwritable note (bad encoding, permissions, disk-full) would raise out of the
  whole render, aborting it mid-run and leaving `note_links` replaced but blocks partially rendered
  (breaks rule 7). **Fixed:** the per-note loop now catches any exception, logs it, counts the note
  `failed`, and continues (skip-and-continue, mirroring the indexer) — with a new test (undecodable
  file → run completes, others still render). Also applied its two hardening MINORs (deterministic
  ordering; index-friendly operand order); MINOR-3 (one-time first-run normalization is not a churn
  bug) documented in the renderer.
- **202 pytest + ruff green** (16 new across `test_graph_renderer` + `test_graph_service`).
  **Verified end-to-end against a real pgvector DB** (dev compose, head 004): LATERAL top-K + floor
  (near-duplicate notes link ~1.0, an orthogonal note is correctly excluded by the 0.5 floor),
  directional `note_links` materialized, blocks rendered only into linked notes, and a **zero-churn
  second run** (churn gate holds).

## Task 6

**Combined nightly `reindex` job + `POST /admin/reindex` async wrapper** — server commits `a059c18`
(feature) + `8cde827` (review minors), local, **not pushed**. The one combined vault-reconciliation
pass (04 §3/§5, [ADR-023](../adr/023-semantic-relatedness-graph.md) §4, 03-api §Admin):

- **`app/services/reindex.py`** — the new `ReindexService`. One pass, two entry points behind a
  **shared single-flight guard**: `git pull` (`vault_backup.sync_from_remote`) → `indexer.reindex_all()`
  (full rescan) → `graph.recompute()` → **one** `vault_backup.backup_now()` (commit+push under the
  single vault lock). `start_manual()` opens the `agent="reindex"` run and runs the pass in the
  **background** (returns the `run_id` for the endpoint's `202`); `run_scheduled()` runs it **inline**
  for the nightly scheduler job and **never raises** (rule 7). `details.trigger` = `manual`/`nightly`;
  index counts (`indexed/skipped/deleted/failed`) + `graph` counts + `commit` result + a top-level
  `partial` land in `details`; the run ends `succeeded` even when `partial` (an embed skip is a
  details flag, not a run status — there is no `partial` status). The **single-flight flag is a plain
  bool**: test-and-set with no `await` between check and set is atomic on the single-threaded loop, so
  a concurrent manual → `None` (→ **409**) and a nightly-while-manual (or vice-versa) → skip; the slot
  is released in a `finally` on every path (including a raise while opening the run row).
- **`VaultBackupService.sync_from_remote()`** — the **pull-first** step, under the one lock: cancel the
  debounce timer + drain pending, abort any stale in-progress merge (never fold conflict markers into
  notes), commit pending local writes so the merge tree is clean, then merge-pull external/other-device
  edits. Best-effort (ADR-014 §5): an unreachable remote / conflicting merge is cleaned up and the
  rescan runs on the current on-disk state — the local vault is never lost. The final `backup_now`
  then carries only the reindex's own graph-block changes (clean "one commit+push for the reindex
  output"; the graph's `content_hash` excludes the block, so it never re-triggers a reindex — the
  feedback-loop fix holds).
- **`POST /admin/reindex`** (`routers/admin.py`) — async: `202 {run_id}`, `409` when a reindex/rescan
  is already running. **Scheduler**: the `reindex` job (03:40, `reindex_cron`) is registered only when
  a `ReindexService` is wired (`BackupScheduler(..., reindex=...)`), so backup-only wiring is unchanged.
  **CLI**: `python -m app.cli reindex` (via `build_reindex_service`) drives the same pass standalone —
  handy for the "DB-wipe + reindex restores search" recovery drill (M2 Accept / Task 9) without the
  authenticated API. Graph recompute stays invoked **only** here (nightly + admin/CLI), never on the
  capture path (grep-confirmed).
- **Independent review (protocol-sanctioned, fresh context): no must-fix.** Verified order + one-lock,
  the race-free single-flight + guaranteed slot release, the `202`/`409` + `agent_runs` contract, rule
  7, and that recompute has a single call site. Applied its clean MINORs (`8cde827`): `sync_from_remote`
  now cancels the timer + drains pending up-front so the day's queued writes land in the pre-pull commit
  with their own reasons (no stale reason on a redundant intermediate push); tightened the shutdown-drain
  comment (a nightly runs in the scheduler executor — best-effort/idempotent on `wait=False` shutdown,
  like the other jobs); de-staled the config schedule comment; added the two missing failure-path tests
  (slot-release when the run-row open raises; `sync_from_remote` aborts an in-progress merge). **Logged
  non-blocking follow-ups:** (1) `reindex_all`'s delete-reconciliation can transiently drop a note
  **captured mid-rescan** (its DB row isn't in the T0 disk scan → deleted as stale), self-healing on the
  next reindex — file never lost (rule 1/2 intact); originates in the Task-3 indexer, and Task 6 is what
  first runs `reindex_all` concurrently with live captures (a daytime `/admin/reindex`), so a defensive
  "re-confirm absent on disk before delete" guard belongs in the indexer if it bites. (2) A
  single-flight-**skipped** nightly leaves no `agent_runs` trace (defensible — matches the daily-summary
  silent-skip pattern, and the concurrent manual run is itself visible).
- **218 pytest + ruff green** (14 new across `test_reindex` + `test_admin_router` + `test_vault_backup`
  + `test_scheduler` + `test_cli`). Covers the pass order, the `agent_runs` contract (trigger/counts),
  partial-flagged-but-succeeds, failure → run `failed` + slot released, the single-flight 409/skip, the
  `202`/`409` router paths, `sync_from_remote` (commit-then-pull / no-op / no-remote / merge-abort), the
  scheduler reindex-job wiring, and the CLI job name + `build_reindex_service`. App boot + OpenAPI
  confirm `POST /api/v1/admin/reindex` is mounted. **Live end-to-end reindex over a seeded vault is the
  M2 Accept (Task 9)** — settings (`RELATED_TOP_K`/`RELATED_MIN_SCORE`, `reindex_cron`) tuned there.

**Organizer tag reuse + `POST /admin/tags/consolidate`** — server commit `b404709`, local, **not

  fail or affect the capture's API response. Cap = `ORGANIZER_TAG_VOCABULARY_MAX` (rule 9).
- **Manual consolidation tool (§2) — `POST /admin/tags/consolidate`.** `app/tags/consolidation.py`

  propose, ≥2 distinct members, deterministic canonical = highest-frequency member, a `seen` set so
  no tag maps to two canonicals), and `rewrite_note_tags` (rewrites **only** the top-level `tags:`
  frontmatter line — inline list or scalar — preserving body / other keys / the `sb:related` block;
  a no-op returns the original bytes verbatim, no churn). `app/tags/service.py`

  one `backup_now` → `202 {run_id}`. Stateless: apply carries the full plan (`plan_id` is a

  apply / 400 apply-without-plan), new wire models, `dependencies`, `main.py` (`PgTagStore` feeds

## Task 8

**Web Search tab + Admin tab** (06 §5/§6). Two decisions surfaced before building (the plan/03-api
didn't settle them; both touch the API contract / ADR-006 — grilled with the user, answered
2026-07-13, recorded in 03-api v2.3):

- **Plane-filter chips need a plane source.** No endpoint returned the configured `PLANES`
  (routers were auth/capture/health/search/admin; `GET /settings` is specced but M4). Options were
  derive-from-results (API-pure but no pre-search chips), hardcode a web constant (drifts, against
  ADR-006), or a small server endpoint. **Decision: add `GET /planes`** — a ~10-line session-gated
  endpoint returning `{ planes, inbox }` from settings, so the web keeps its only-knows-the-API
  contract and gets the spec'd pre-search scoping chips.
- **"Reindex shows the run's live counts" but nothing could read a run.** `POST /admin/reindex` and
  tags-**apply** return only `202 {run_id}`; `GET /activity/runs/{id}` was M4/unbuilt. **Decision:
  build the minimal `GET /activity/runs/{id}` now** (pulled forward from the M4 feed) — one
  `agent_runs` row (status + `details` counts) via a new `AgentRunStore.get`; the merged `GET
  /activity` list stays M4. Admin polls it for live `indexed/skipped/deleted/failed`.

**Server (two small endpoints).** `AgentRunStore` gained `get(run_id)` (+ a `_row_to_run` helper
