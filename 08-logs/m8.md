## Task 1 — Observability foundation

**The one genuinely new M8 subsystem (live per-run log capture) + the JobRunner single-flight
seam + the sole M8 migration.** Solo task, first in the milestone; **Batch B {T2,T3,T4}** fans out
after it (they need `trigger`, the JobRunner, and the run-scope this task lands). Server-only.
**Committed locally (`4750f12`), not pushed.**

### What was built

- **Migration 015** (`agent_run_logs` + `agent_runs.trigger`, [ADR-053](../adr/053-m8-ops-console-observability-build-decisions.md)
  §1/§2/§5) — additive, reversible, plain SQL (ADR-011). `agent_run_logs(id bigserial, run_id uuid
  FK→agent_runs ON DELETE CASCADE, seq int, ts timestamptz, level text, message text)` + a UNIQUE
  `(run_id, seq)` index that serves the `seq > after_seq ORDER BY seq` poll and guards a double
  flush. `agent_runs.trigger text NOT NULL DEFAULT 'scheduled'` (existing rows backfill correct —
  they were all scheduler/CLI-driven). Both **rebuildable op-state, not graph truth** (rule 1).
- **`app/services/run_logs.py`** — the capture subsystem. A process-wide `RunLogHandler`
  (`logging.Handler`) filters to **`app.*` at `INFO`+** (namespace + explicit level check, so
  library `DEBUG` — the secret-leak surface — is structurally kept off the UI-rendered store, rule
  11), reads the **active run** from a `_current_run_id` contextvar, and appends
  `(seq, ts, level, message)` to a **bounded per-run in-memory buffer** synchronously (never awaits
  a DB write, rule 8). Overflow **drops-oldest and emits a `… N lines elided` marker** (rule 7 — no
  silent cap). `seq` is **1-based** so the poll's default `after_seq=0` cursor includes the first
  line. A `RunLogFlusher` persists new lines to `agent_run_logs` on a **~1s cadence + immediately on
  run finish** (then reaps the buffer, so a long-lived process doesn't accumulate one buffer per
  historical run); a flush failure logs and drops the batch, never crashing the loop (rule 1).
  `PgRunLogStore` is the plain-SQL read/write.
- **Run scope in `agent_runs`** — the log tagging is owned by the store (it mints the run id), not
  job bodies (ADR-053 §7): `start` pushes the run id onto an **immutable-tuple contextvar stack**
  (`begin_run_scope`), `finish` pops it + fires the run-finish hook (`end_run_scope`). Task-safe
  (each task/`to_thread` worker carries its own stack) and correctly **nested** — a pipeline
  parent's own lines tag the parent while a step's lines tag the step's child run. Plus the
  **`_trigger` contextvar** (`current_trigger`/`trigger_scope`) `start` stamps onto the row, and an
  optional **run-finish hook** (`set_run_finish_hook`) the flusher wires for the on-finish flush.
- **`app/services/job_runner.py`** — the in-process **single-flight guard** the scheduler and the
  manual endpoints share (ADR-053 §7; authoritative — the scheduler runs single-process). A
  running-set mutated only between awaits on the one loop. `run_manual` (T3 will use) takes the slot
  or raises `JobAlreadyRunning` → **409**, and wraps the call in `trigger_scope(MANUAL)`;
  `scheduled_step` (the pipeline runner uses) advisory-marks a step so a manual trigger of the same
  agent 409s, and **skips** the step if a manual run already holds it (idempotent, rule 6). Threaded
  through `PipelineScheduler` → each `PipelineRunner` (optional; the CLI run-now passes none).
- **`store-sweep` own run row** (ADR-053 §10) — `run_store_sweep` now opens/closes its own
  `agent_runs` row (rule-7 wrapped, records `committed`/`pushed`), killing the phantom `skipped`
  pipeline step. Needs no `object_store` (it's a git commit+push), so it runs in dev too.
- **`GET /activity/runs/{id}/logs`** — the live log tail (03-api §Activity): poll (`?after_seq=` +
  a `running` flag so the client stops when the run isn't active), 404 on unknown / 422 on malformed
  id or negative cursor, session-gated. `AgentRunResponse` gains `trigger`. Config knobs
  (`run_log_buffer_max_lines`, `run_log_flush_interval_seconds`, `run_log_tail_max_lines`, rule 9).
- **Wiring** (`main.py`) — the buffers/handler/flusher + `JobRunner` are built in the lifespan,
  the handler attached to the root logger, the finish hook registered, the flusher started; teardown
  clears the hook, detaches the handler, and does a final flush **before** the pool closes. All on
  `app.state` so Batch B reads them without re-editing `main.py`.

### Review + verification

- **764 unit tests green** (+43: buffer seq/overflow/floor, handler namespace+level+scope+nesting,
  flusher cadence/on-finish-reap/shutdown-drain/error-survival, JobRunner all branches + the
  pipeline single-flight-skip integration, the logs endpoint running-flag/cursor/404/422), ruff +
  format clean.
- **Migration 015 up/down/up verified** on local pg; **real-PG smoke +7** (`check_run_logs_and_trigger`:
  trigger default/manual/reset, log round-trip in seq order, `read_after` cursor + limit, ON CONFLICT
  idempotency, FK cascade on run delete) → **136/136**.
- **End-to-end verified** against real pg: a real `PgAgentRunStore` run → `app.*` INFO lines captured
  and tagged to the run (`scanning store`, `indexed 3 nodes`), a `DEBUG` "secret" line **and** a
  `uvicorn.access` line **filtered out**, seq 1-based, and a line emitted **after** finish (outside
  any run scope) **not** captured.
- **Independent review APPROVE — no must-fix.** The reviewer confirmed the contextvar stack is
  task-safe + correctly nested, the flusher/shutdown ordering is sound, seq/cursor semantics are
  right, the single-flight guard is race-free on the one loop, the migration is additive/reversible,
  and the rule-11 filter is structurally robust. One nit applied (tightened the `ON CONFLICT` comment
  to not overstate a retry path the live design can't reach). **Follow-ups logged (for Batch B / T5):**
  the on-finish flush is *scheduled* not awaited, so a client that stops **solely** on `running==false`
  can miss the last <1s tail + must page past `run_log_tail_max_lines` — **the web log-tail client
  (T5) must page until an empty page, not stop on the flag alone** (03-api client guidance);
  the run-stack pop is coupled to `finish` (a job that starts a run and never finishes it — DB-down —
  leaks a stack entry, bounded, value-filtered pop prevents corruption).
