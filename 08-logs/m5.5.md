Per-task build narratives for **M5.5** — what was built, the independent review, and the

[../08-implementation-plan.md](../08-implementation-plan.md) (§M5.5). This file is append-only

## Task 1

**Pipeline runner + config model + `agent_runs.parent_run_id` migration + parent/child linkage**
([ADR-047](../adr/047-pipeline-scheduling-primitive.md)) — the orchestration **mechanism** only;
**no scheduler wiring, no job migrated** (that is task 2). **Local, not pushed.** Commit `c0a3bd6`.

The pipeline is the scheduling primitive: a name + one cron + an ordered list of steps run
sequentially, each starting only when the previous completes. Task 1 builds the runner, the config
model, the DB linkage, and unit tests **with fake steps** — deliberately without touching the live
`BackupScheduler` or any job, so the mechanism is proven in isolation before task 2 hangs the real
roster on it.

**Migration 012** (`server/migrations/versions/012_agent_runs_parent_run_id.py`, plain SQL / no
ORM, rule 5). Adds a nullable, self-referencing `agent_runs.parent_run_id uuid REFERENCES
agent_runs(id) ON DELETE SET NULL` + a `parent_run_id` index. A bare job run — the standalone CLI /
`POST /agents/{name}/run` path (invariant 4) — leaves it `NULL` exactly as before, so the existing
per-job observability is unchanged (ADR-047 §5); only a step run under a pipeline links to a parent.
`ON DELETE SET NULL` so purging a parent never orphans a child into a dangling FK. Up/down
round-trip verified on local pg (column + FK `delete_rule=SET NULL` + index present after upgrade,
all gone after downgrade, DB re-pinned at head).

**Parent/child linkage via contextvars** (`server/app/services/agent_runs.py`). The load-bearing
constraint is ADR-047's "**no job changes what it does**": a job's `run_scheduled` keeps calling
`start(AGENT)` with no idea it runs under a pipeline. So linkage is **ambient**, not a passed
argument — a `child_run_scope(parent_run_id)` context manager (runner-side) sets a task-local
`ContextVar` that every store's `start` reads (`current_parent_run_id`) to stamp `parent_run_id`,
and appends the new run id to the scope's collector (`record_child_run`) so the runner captures
exactly the children a step opened. The scope is task-local and token-reset on exit, so a concurrent
manual run firing **outside** the scope is never miscaptured as a child. `parent_run_id` threaded
through the `AgentRun` dataclass + the `PgAgentRunStore` INSERT + both SELECTs (`latest`/`get`); the
`GET /activity/runs/{id}` response is **unchanged** (surfacing the parent link is deferred to the M8
ops console, ADR-047 §5).

**The runner** (`server/app/services/pipeline.py`). `PipelineStepDef` (name + `on_fail`,
validated `continue`|`halt`) + `PipelineDef` (name + one cron + ordered non-empty steps) are the
config model; `PipelineRunner` opens a **parent** `agent_runs` row, runs each step inside
`child_run_scope`, and — because a job never raises and returns `None` ambiguously — **infers each
step's status by reading its child run back**: a child that is not *cleanly done* (`failed`, an
orphaned `running`, or an unknown status) reads as a **failure**, so a `halt` step reliably aborts
the rest rather than proceeding on half-built state (ADR-047 §4); a step that opened no child at all
(job couldn't reach the DB) is `skipped` (not a failure); a `missing` runnable (definition/wiring
mismatch) is a failure that still honours `on_fail`. A `continue`-step failure is recorded per-step
and the pipeline proceeds (the rule-7 default — one flaky call never costs the night its backups); a
`halt`-step failure aborts and the parent run closes `failed` (`halted_at`). The parent's `details`
records the full per-step sequence + status. Defensive rule-7 hardening: if a job **does** raise, the
runner closes the child row it left `running` as `failed` (no orphaned `running` row) and never
crashes the scheduler; if the parent row can't even be opened (DB down), `run()` returns `None` and
logs.

**Config model** (`server/app/config.py`). `nightly_pipeline_cron` (`0 3 * * *`) +
`weekly_pipeline_cron` (`30 4 * * sun`, byte-identical to the retired `integrity_drill_cron` — no
schedule regression) + `pipeline_defs()` returning the two definitions: the **nightly** roster is the
migrated ADR-010 order in dependency order (`data-sync → db-backup → reindex → profile-refresh →
entity-backfill → identity-capsule-refresh → store-sweep → store-backup`), the **weekly** pipeline is
the `integrity-drill`. **`continue`-dominant** (ADR-047 §4): no migrated durability step is a
foundational precondition that should abort the rest, so all are `continue` — the `halt` policy
exists and is unit-tested, but the roster doesn't need it. Step names map exactly to the existing
`BackupScheduler` job ids, so task 2's name→func wiring is a lookup. A lazy import inside
`pipeline_defs()` breaks the `config → pipeline → agent_runs → db → config` import cycle.

**Tests.** 16 unit tests (`tests/test_pipeline.py`, fake steps: sequential ordering,
`continue`-vs-`halt`, parent opened + children link back + parent stays parentless, running-child →
failed → halt, raising-step orphan close, missing runnable, no-run → skipped, parent-open failure →
`None`, definition validation, the config roster shape + crons parse) + the shared `FakeAgentRunStore`
updated to honour the same contextvar linkage so the mechanism is exercised identically in tests. Plus
**6 real-PG smoke checks** (`scripts/smoke_db.py`) for the un-fakeable `parent_run_id` INSERT + FK
round-trip + contextvar reset: bare run parentless, scope collects both child ids, both children link
to the parent, parent stays parentless, a start after the scope is parentless again. **619 unit tests
+ 79 real-PG smoke green, ruff clean.**

**Independent review — APPROVE, no must-fix.** A fresh agent re-derived Task 1's scope from ADR-047 +
§M5.5 and checked the diff: confirmed sequential-on-completion, `on_fail`, the contextvar linkage
(task-local, reset, no leak), the migration chain (011→012, no gap), the roster names matching the
scheduler ids, the weekly cron matching the retired one, and **zero behavior change today** (scheduler
untouched — the durability drill path is unaffected). **4 minors applied:** (1) robust `halt`
inference — a child left `running` now reads as a failure, not success; (2) a raising step closes its
orphaned child run `failed` (rule 7); (3) the `SKIPPED` constant instead of a `"skipped"` literal; (4)
a skipped count in the summary line. **Noted for task 2** (not task-1 defects): the step-name→func map
must omit a **disabled/optional** job (reindex/profile-refresh/identity-capsule are feature-gated in
the scheduler) so a disabled step isn't recorded `missing` every night; register the pipeline cron
with the same `max_instances=1`/`coalesce` self-overlap guard; the nightly/weekly Sunday overlap is
pre-existing (the retired drill cron was the same slot), flag for conscious acceptance.

**Next: M5.5 task 2** — migrate the roster off individual crons into the two pipelines (scheduler
registers one cron per pipeline; CLI/run-now entrypoints unchanged), retire the per-job cron settings,
verify the durability drill + a manual run-now mid-pipeline.

## Task 2

**Migrate the nightly roster off per-job crons into the two pipelines**
([ADR-047](../adr/047-pipeline-scheduling-primitive.md) §6/§7) — **orchestration only, no job body
changed.** **Local, not pushed.** Commit `8a57611`.

The in-process scheduler now registers **one cron per pipeline**, not one per job.
`BackupScheduler` → **`PipelineScheduler`** (`server/app/services/scheduler.py`): it builds a
**step-name → job-coroutine map** (the five backup jobs always wired; `reindex` / `profile-refresh` /
`entity-backfill` / `identity-capsule-refresh` present only when their service was constructed — they
stay optional exactly as before), constructs **one `PipelineRunner` per pipeline def**
(`Settings.pipeline_defs()`), and registers each pipeline's single cron with the same ADR-010 wiring
(`max_instances=1` + `coalesce` + misfire grace) so a long night can't overlap the next fire (the RAM
guarantee the stagger used to give, ADR-047 §3). An **unwired optional step is dropped with a log**
(rule 7, no silent cap) so it's never scheduled as a doomed `missing` step; a pipeline left with no
wired steps is skipped. `pipeline_runners()` is pure (returns `(def, runner)` per pipeline), so the
mapping unit-tests without a loop. `main.py` constructs it with the real `PgAgentRunStore` so the
parent run opens and children link (ADR-047 §5). **CLI (`app/cli.py`) + `POST /admin/reindex`
standalone-invoke paths untouched** (invariant 4 / §6).

**Retired the nine per-job `*_cron` settings** (`reindex_cron`, `backup_data_sync_cron`,
`backup_db_backup_cron`, `integrity_drill_cron`, `profile_refresh_cron`, `backfill_cron`,
`identity_capsule_refresh_cron`, `backup_store_sweep_cron`, `backup_store_bundle_cron`) — dead once
nothing schedules per-job (grep confirms zero remaining references in `app/`). Kept
`nightly_pipeline_cron` (`0 3 * * *`) / `weekly_pipeline_cron` (`30 4 * * sun`, **byte-identical** to
the retired `integrity_drill_cron` — no schedule regression) + `integrity_drill_max_age_days`. The
nightly roster maps **1:1** onto the old stagger in the **same order** (data-sync → db-backup →
reindex → profile-refresh → entity-backfill → identity-capsule-refresh → store-sweep → store-backup);
no old job dropped or duplicated.

**Verification.** 619 unit tests + 79 real-PG smoke green, ruff clean. `test_scheduler` rewritten for
one-cron-per-pipeline (runner mapping, step→job wiring, unwired-step drop → backup-only nightly,
whole-pipeline skip, real `start()` registers exactly `{nightly, weekly}` + misfire/overlap wiring). A
throwaway **real-PG integration** drove the full wiring end-to-end (real `PipelineScheduler` + real
`BackupJobs` over git/R2 fakes + real `PgAgentRunStore`): the nightly `PipelineRunner` opened **one
parent `nightly` run**, each row-opening step (`data-sync`/`db-backup`/`store-backup`) linked a child
by `parent_run_id` **in order**, the pipeline completed (continue-dominant), and **`store-sweep` ran
but opened no `agent_runs` row — unchanged behaviour**. The **DB-wipe → reindex durability drill** is
preserved by construction: the diff touches only scheduler/config/main/tests (no job body), and the
`reindex` step drives the identical `ReindexService.run_scheduled`; the live drill re-run belongs to
**task 3**.

**Independent review — APPROVE, no must-fix.** A fresh agent re-derived Task 2 scope from ADR-047 +
§M5.5 and checked the diff: one cron per pipeline with the overlap guards, no job body changed, the
nine retired crons have zero dangling references, the drop-logic is prod-safe (main.py wires all four
optional jobs unconditionally, so all 8 nightly steps always run in prod), the old→new roster maps 1:1
with order preserved, and the real `run_store` links parent/child. **One minor logged (no code
change):** on **Sundays** `nightly` (03:00) and `weekly` (04:30) are independent APScheduler jobs, so a
long nightly overrunning to 04:30 fires the heavy integrity-drill **concurrently** — two steps' RAM at
once. This is a deliberate **ADR-047 §3** two-pipeline residual (strictly better than the old per-job
design; store-git work still serialises on the one lock so only peak RAM, not correctness, is at
risk), not a task-2 regression — flag if the VPS ever OOMs on a Sunday. **Second follow-up:**
`run_store_sweep` records no `agent_runs` row of its own (a pre-existing lightweight commit+push), so
under the pipeline it appears as a **`skipped`** step in the parent's step list; giving it its own row
= changing a job = out of M5.5's orchestration-only scope — a candidate polish for the M8 ops console.

**Next: M5.5 task 3** — live M5.5 Accept (deploy; confirm one nightly start drives the whole roster in
order, per-step runs visible under the parent, durability drill green) → independent review → pause.

## Task 3 — live M5.5 Accept → **M5.5 CLOSED (2026-07-16)**

**Deployed to prod + live-verified.** Pushed `2dc8f92..e77a694` (tasks 1+2 + a `pipeline` run-now
verb); CI green, migration 012 applied on prod, `/api/v1/health` all-true throughout. **Commits
`c0a3bd6` / `8a57611` / `e77a694`** pushed; **`2f6c8fb`** (review fidelity fix) committed local.

**The run-now verb (`e77a694`, an ADR-047 §6 enabler).** There is no on-demand pipeline trigger and
prod `agent_runs` can't be observed from outside the authenticated session, so waiting for the 03:00
cron was the only other way to verify. Added **`python -m app.cli pipeline {nightly|weekly}`**: it
builds every step's service from the existing `build_*` helpers, borrows `PipelineScheduler`'s runner
wiring, drives the chosen `PipelineRunner` to completion, and **prints self-contained evidence**
(parent run id + each step's status + child run id). `PipelineOutcome` gained `parent_run_id`. Per-job
CLI + `POST /admin/reindex` untouched.

**Live Accept — PASSED.** The user ran `python -m app.cli pipeline nightly` on the VPS. One start
drove **all 8 steps in dependency order** under a single parent run `939306a4-...`:
`data-sync → db-backup → reindex → profile-refresh → entity-backfill → identity-capsule-refresh →
store-sweep → store-backup` — **7 succeeded, 0 failed, 1 skipped**, each row-opening step carrying a
distinct `child_run` id linked to the parent. **`store-sweep` shows `skipped/None`** — the pre-logged
follow-up (`run_store_sweep` opens no `agent_runs` row of its own; its log line `committed=False
pushed=True` proves it **ran**). Every job did its normal nightly work (reindex: 600 similar edges +
`pushed=True`; profile-refresh: 12 regenerated of 75; capsule refreshed 1599 chars; store-backup
pushed), confirming **no behavior change**. This single run also **confirms migration 012 applied on
prod** — the parent+child `parent_run_id` INSERT succeeded, which is impossible without the column.

**Accept criteria — all met:** every nightly job runs as a **step in a scheduled pipeline** (not its
own cron); `nightly`/`weekly` each have **one start time**; steps run **sequentially on completion**
(timestamps: reindex done 07:44:40 → profile-refresh embeds → … → store-backup 07:46:09); `continue`
vs `halt` honoured (unit-tested; the roster is continue-dominant); each step opens its **own child
`agent_runs` under a parent**; any step is still **runnable standalone** (per-job CLI + `POST
/admin/reindex` intact); **no behavior change** — the DB-wipe→reindex durability drill is preserved by
construction (no job body changed; the `reindex` step ran healthy). *Note:* the run captured a
**steady-state** reindex (`0 indexed, 120 skipped, 1 deleted`) — the full wipe→restore drill (proven
byte-identical in M3) wasn't re-run since no job changed; the reindex step executing + pushing inside
the pipeline covers the orchestration risk.

**Independent review — APPROVE, no must-fix.** A fresh agent checked the `e77a694` diff + the live
evidence against the Accept criteria + ADR-047: the roster runs in order under one parent, the
migration-012 inference is airtight, arg-parsing is backward-compatible, `PipelineOutcome.parent_run_id`
is additive (no shape break), and store-sweep's `skipped` is the known unchanged behavior. **One
worthwhile minor fixed (`2f6c8fb`):** the run-now verb built profile-refresh/backfill **seeds-only**
(`vocab=None`), so it could scan a narrower entity set than the in-app nightly if a governance
entity-like type were approved — now it threads a real `VocabularyService` provider (seeds ∪ approved
additions) so run-now faithfully reproduces the cron (`VocabularyService`'s review/consolidation deps
default to `None` so it builds as a pure provider; the two builders gained an optional `vocab` param,
default `None` preserving the per-job CLI). Also a fast-fail on an unknown pipeline name before DB
setup. No functional impact on the Accept (prod has no approved entity-like additions), so the passed
run stands. Other minors logged, not fixed (all pre-existing / out of orchestration scope):
`run_store_sweep` opens no row (→ `skipped` in the parent view); the per-job CLI `profile-refresh`/
`entity-backfill` remain seeds-only; a continue-only all-fail nightly still exits 0 (parse the per-step
lines, ADR-047 §4); the full wipe→reindex drill not separately re-run.

**M5.5 CLOSED** — the pipeline is the scheduling primitive; the whole nightly roster runs sequentially
from one start under a parent `agent_runs` run with per-step children, live on `braindan.cc`.
